# D-016 — service_type: видалити custom, додати other з ручним описом

**Дата:** 2026-05-17

**Рішення:**

1. Enum `service_type` (на `Payment` і `Act`) змінюється з `access / sms / custom` на `access / sms / other`.
2. Додається поле `Act.service_description` (string) — human-readable опис послуги, що рендериться в PDF акту.
3. Для `service_type = other` логіка така:
   - Присвоюється **тільки вручну** адміном через UI (автокласифікація ніколи не повертає `other`).
   - `unit_price`, `quantity`, `quantity_unit` задаються вручну.
   - `service_description` обов'язкове.
   - Інваріант з D-010 зберігається: `unit_price × quantity == payment.amount`.
4. Для `access`/`sms`: `service_description` генерується автоматично за шаблоном при створенні акту (наприклад, "Доступ до сервісу за період 1 місяць", "СМС-розсилка 100 шт"). Адмін може відредагувати перед відправкою в EDO.

**Альтернативи:**

- Видалити `custom` без заміни (YAGNI). Відкинуто: реальні випадки існують (нестандартні платежі від Авалон, Урбан тощо), для яких треба явна позначка "це не стандартна послуга" — інакше в звітах і фільтрах вони змішуються зі звичайними access/sms актами.
- Лишити `custom` з тим самим змістом, що і `other`. Відкинуто: слово `custom` натякає на "configurable", `other` чітко каже "fallback для нестандарту".
- Створити окремі enum значення під кожен нестандартний кейс (`avalon_consulting`, `urban_setup` тощо). Відкинуто як надмірно гранульоване — модель тоді росте при кожному новому клієнті.

**Обґрунтування:**

- `other` ортогональний до `auto_act_disabled` (D-003): перший — про природу послуги, другий — про процес обробки. Один клієнт може мати кілька актів різних типів.
- Дозволяє адміну позначити нестандартні акти в UI, фільтрувати/групувати їх у звітах і знати, що для цього платежу не діють стандартні правила тарифу.
- `service_description` уніфікує рендер PDF для всіх типів: автогенерація для access/sms, ручне введення для other.

**Наслідки:**

- `Payment.service_type` і `Act.service_type` enum: `access / sms / other`.
- Нове поле `Act.service_description` (string, not null).
- Алгоритм класифікації (крок 4) не змінюється — повертає тільки `access` або `sms`.
- В адмінці (PRD 8.2): на картці платежу/акту з'являється можливість перемкнути `service_type` на `other` (доступно тільки для платежів у статусах `awaiting_review` або `in_queue`, не для вже `classified` з відправленим актом).
- При перемиканні на `other`: розблоковуються поля `unit_price`, `quantity`, `quantity_unit` для ручного редагування; `service_description` стає обов'язковим.
- Шаблони актів (PRD 6.3): додається шаблон "other" з кастомним `service_description`.
- В `Settings` НЕ додаються правила автокласифікації для `other` — за дизайном.
- Запис у `CHANGELOG.md`, версія `domain_model.md` → 0.3 (мінор, бо новий enum value + нове поле).

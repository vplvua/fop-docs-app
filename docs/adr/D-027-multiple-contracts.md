# D-027 — Кілька номерів договору в одному purpose → in_queue(multiple_contracts)

**Дата:** 2026-05-18

**Рішення:**

1. Додається нове значення enum `classification_reason`: `multiple_contracts`. Належить до `Payment.status = in_queue`.
2. Після парсингу `parsed_contract_numbers` виконується **dedup** (унікальні значення).
3. Якщо після dedup залишилось `> 1` різних номерів — платіж потрапляє в `in_queue(multiple_contracts)`. Класифікатор НЕ намагається обрати "правильний" номер автоматично.
4. Якщо після dedup залишилось рівно `1` номер — продовжуємо звичайний матчинг.
5. Якщо `parsed_contract_numbers` порожній після dedup — fallback на ЄДРПОУ (як було).
6. В UI 8.3 (Черга розбору) платіж з `multiple_contracts` показує всі знайдені номери і дозволяє адміну обрати один (або позначити `skipped`, якщо платіж реально стосується кількох клієнтів і потребує неавтоматизованого процесу).

**Альтернативи:**

- **Брати перший знайдений (`parsed_contract_numbers[0]`).** Відкинуто: результат залежить від порядку патернів у налаштуваннях і порядку входжень у тексті. Перестановка patterns у Settings може непомітно змінити класифікацію. Тихі неправильні класифікації — гірше за явні `in_queue`.
- **Якщо всі номери ведуть до того самого Client → класифікувати, інакше in_queue.** Відкинуто: не покриває реальний сценарій "оплата за дві ОСББ одним платежем" — два різних `Client.id`, обидва матчаться, акт буде створений на випадково першого. D-007 (1 платіж = 1 акт) робить такий випадок принципово некласифіковним без розщеплення — а розщеплення поза скоупом (D-007).
- **Створити кілька актів для одного платежу.** Відкинуто: суперечить D-007 і ускладнює модель Payment ↔ Act з 1:1 на 1:N.

**Обґрунтування:**

- Силова класифікація з кількох кандидатів — клас "мовчазних помилок", які важко помітити постфактум.
- Реальний сценарій з виписки (`research/payments-analizys/analiz_nadhodzhen.md`): платник може помилково/свідомо вписати два номери ("За договорами №X і №Y"). Жоден з варіантів автокласифікації не безпечний.
- `multiple_contracts` як окремий reason дає чітку індикацію в UI: адмін одразу розуміє, що проблема не "клієнта не знайшли", а "знайшли неоднозначно".

**Наслідки:**

- В моделі: enum `classification_reason` → `+multiple_contracts`. ENUM type у PostgreSQL: `ALTER TYPE classification_reason ADD VALUE 'multiple_contracts'`.
- Псевдокод кроку 2 (матчинг клієнта) оновлюється: dedup + перевірка `len(parsed_contract_numbers) > 1`.
- В UI 8.3 — нова секція або фільтр для `multiple_contracts`; картка платежу показує знайдені номери як обираний список.
- В PRD 4.2 (Парсинг номера договору) і 4.6 (Причини попадання в чергу) додається відповідний пункт.
- Версія `domain_model.md` → 0.9 (мінор: нове enum value + новий логічний крок).

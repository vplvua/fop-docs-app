# D-022 — Розширити client_incomplete: bank_name і bank_account обов'язкові

**Дата:** 2026-05-17

**Рішення:** У крок 5 логіки класифікації (`client_incomplete` check) додаються два нових обов'язкових поля клієнта:

- `bank_name` — назва банку клієнта;
- `bank_account` — IBAN клієнта.

Якщо хоча б одне порожнє — `in_queue(client_incomplete)` з відповідними значеннями в `missing` (наприклад, `missing=["bank_account"]`).

**Альтернативи:**

- Робити обов'язковим лише `bank_account` (без `bank_name`). Відкинуто: у PDF акту обидва поля рендеряться як один рядок ("Поточний рахунок: UA... в АТ КБ ..."), без назви банку рядок неповний.
- Зробити умовно обов'язковими (наприклад, тільки для ОСББ). Відкинуто: будь-який платник, чий платіж проходить через нашу систему, апріорі має рахунок з якого зробив переказ — універсальне правило простіше.
- Об'єднати в одне missing-значення `bank` замість двох (`bank_name`, `bank_account`). Відкинуто: адміну корисно бачити, чого саме не вистачає — назви банку чи IBAN.

**Обґрунтування:**

- Зразок акту (`research/act-sample/act-2026-04_200.pdf`) явно показує bank_name + bank_account як обов'язкові реквізити Замовника.
- Поля вже існують в моделі (`Client.bank_name`, `Client.bank_account`), просто не перевірялись на повноту.
- Узгоджує set обов'язкових полів з реальною формою акту.

**Наслідки:**

- В кроці 5 псевдокоду додаються 2 нових перевірки.
- Запис у `client_snapshot` JSON структурі вже містить bank_name/bank_account — без змін.
- Сам опис полів `Client` лишається `nullable` на рівні БД (бо локальний клієнт без moeosbb може бути створений неповним і доукомплектовуватись), але класифікація платежу блокується без них.
- Версія `domain_model.md` → 0.6.3 (patch — додавання двох умов в існуючий крок).

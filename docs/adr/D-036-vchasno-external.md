# D-036 — Зовнішнє ЕДО (Вчасно): vchasno_external як другий канал оформлення актів

**Дата:** 2026-05-19

**Рішення:** Модель починає розрізняти два канали оформлення акту — автоматичний через Дубідок API і ручний з підписом у зовнішньому ЕДО (Вчасно). Розширення:

1. На `Client` додається поле `edo_provider` — enum (`dubidoc`, `vchasno_external`), default `dubidoc`. Одне значення на клієнта в будь-який момент часу; зміна провайдера = свідома міграція, історичні акти не переоформлюються.
2. На `Act` додається snapshot-поле `edo_provider` того ж enum-типу, копіюється з клієнта в момент створення акту (за аналогією з `client_snapshot`, `contract_snapshot`).
3. `classification_reason` отримує нове значення `external_edo` — причина потрапляння платежу в `awaiting_review` для клієнтів з `edo_provider = vchasno_external`.
4. Логіка класифікації після кроку 3 (перевірка `auto_act_disabled`) отримує крок 3b: якщо `client.edo_provider = vchasno_external`, повертаємо `awaiting_review(reason=external_edo)`. Прапорці `auto_act_disabled` та `edo_provider` — ортогональні; при одночасній установці пріоритет має `auto_act_disabled` (бо вказує на потенційну проблему з даними, що важливіше за канал ЕДО).
5. State machine `Act` диференціюється за `edo_provider`:
   - `dubidoc`: `draft → sent_to_edo → signed` (або `→ deleted`) — без змін.
   - `vchasno_external`: `draft → signed`. Переходи `sent_to_edo` та `deleted` не застосовні (API-виклику немає, видалення ззовні нам невідоме).
6. Для `vchasno_external` поля `edo_doc_id` та `edo_status` залишаються NULL назавжди (це поля polling-а Дубідок).
7. Перехід `draft → signed` для `vchasno_external` виконується окремою дією адміна "Позначити підписаним" (немає джерела істини зовні, яке ми могли б опитувати — це явне ствердження адміна).
8. PDF для `vchasno_external` актів можна перегенерувати без обмежень навіть після `signed`. Канонічна підписана версія живе у Вчасно; локальний PDF — допоміжна копія для аудиту.

**Альтернативи:**

- Залишити одне поле `auto_act_disabled` і трактувати "клієнт хоче Вчасно" як ще один варіант ручного оформлення. Відкинуто: `auto_act_disabled` семантично означає "дані нестандартні, треба ручна перевірка" (D-003), а тут проблема не з даними, а з каналом ЕДО — змішування блокує осмислену фільтрацію в UI.
- Об'єднати обидва прапорці в один enum `auto_mode` (`auto_dubidoc / manual_dubidoc / manual_vchasno`). Відкинуто: два кейси ортогональні (один клієнт може мати обидві ознаки), а enum змусив би вибрати одну, втрачаючи інформацію.
- `edo_provider` на рівні `Payment` (вибір при класифікації). Відкинуто: ФОП не змішує канали для одного клієнта — рішення про ЕДО приймається з клієнтом одноразово на початку взаємодії.
- Окремий термінальний статус `signed_externally` замість загального `signed`. Відкинуто: семантично "акт підписаний" — один стан, дискримінувати "де саме" має `edo_provider`, не `status`.

**Обґрунтування:**

- Бізнес-кейс: частина клієнтів категорично проти Дубідок або має внутрішні вимоги до конкретного ЕДО (Вчасно). Власник готовий обробляти таких клієнтів вручну.
- D-015 явно передбачив цей крок як майбутнє рішення: "якщо такий use case виникне — окреме рішення про додавання `edo_provider` поля на акт/контракт".
- Snapshot на акті потрібен з тих самих причин, що `client_snapshot` / `contract_snapshot`: акт оформлений як Вчасно-документ має лишатись таким навіть якщо клієнт згодом мігрує на Дубідок.
- Ортогональність двох прапорців відображає реальність: ОСББ може одночасно мати нестандартний договір (вимагає ручної перевірки розрахунку) і не використовувати Дубідок. Об'єднання в один enum приховало б одну з ознак.
- PDF без заморозки після `signed` — бо локальний PDF не є каноном. Симетрію з Дубідок-актами (де PDF заморожений після `sent_to_edo`) можна не тримати, оскільки в Дубідок-кейсі канонічна версія лежить в EDO і має збігатись з тим, що ми відправили.

**Наслідки:**

- `domain_model.md`:
  - `Client`: нове поле `edo_provider` enum default `dubidoc`. Origin: `manual` (sync з "Моє ОСББ" не зачіпає, як і `auto_act_disabled`).
  - `Act`: нове snapshot-поле `edo_provider`.
  - Новий PostgreSQL ENUM: `edo_provider AS ENUM ('dubidoc', 'vchasno_external')`.
  - `classification_reason`: додається значення `external_edo`.
  - Логіка класифікації: новий крок 3b після перевірки `auto_act_disabled`.
  - State machine `Act`: дві гілки залежно від `edo_provider`.
  - Розділ "Інтеграції" з зауваженням: Вчасно — без API-інтеграції, тільки ручний експорт PDF.
- PRD:
  - Розділ про адмінку: бейдж "Вчасно" / "Дубідок" в переліку актів, фільтр по `edo_provider`.
  - Дія "Позначити підписаним" для актів з `edo_provider=vchasno_external`.
  - Перевипуск/перегенерація PDF дозволені для vchasno_external навіть у статусі `signed`.

# D-033 — Розділ "Договори" в адмінці: повноцінний CRUD з редагуванням після генерації актів

**Дата:** 2026-05-18

**Рішення:**

1. Розділ 8.5 PRD ("Договори") — **повноцінний CRUD**: створення, перегляд, редагування, видалення (з обмеженнями).
2. **Редагування договору після того, як на нього вже згенеровані акти — дозволено**. Snapshot номера і дати в `Act.contract_snapshot` (D-019) захищає історичні акти від змін.
3. Видалення договору блокується БД, якщо на нього є акти (ON DELETE RESTRICT — D-025). Адмін отримує помилку валідації; пропонується створити новий договір замість видалення.
4. У формі редагування — попередження: "Зміна номера/дати договору не торкне підписаних актів — там зберігається snapshot. Нові акти будуть генеруватись із новими даними."

**Альтернативи:**

- **Read-only довідник** (як було в скелеті). Відкинуто: суперечить D-019, де договір обов'язковий перед актом — адмін мусить мати спосіб його створити з UI. Якщо передбачали "договори імпортуються з зовнішньої системи" — в реальності такої системи немає.
- **Create + Read, edit заблокований після першого акту.** Відкинуто: snapshot-патерн (D-004, D-010, D-019) спеціально розроблений для того, щоб дозволити правки в master-даних без втрати юридичної цілісності історичних документів. Блокування редагування — це паніка-захист, який не потрібен у цьому дизайні.

**Обґрунтування:**

- Snapshot — це і є архітектурна гарантія "минулі акти не псуються". Якщо ще й блокувати редагування — це redundant constraint.
- Реальний кейс: адмін помилково ввів номер договору з опечаткою, на нього вже згенеровано акт. Виправляти треба і поле, і одночасно перевипустити акт через нову класифікацію — snapshot старого акту лишається з опечаткою (як було підписано), новий генерується з правильним.

**Наслідки:**

- В PRD 8.5 переписується: CRUD з warning'ом про snapshot, RESTRICT на delete.
- В моделі — без змін.

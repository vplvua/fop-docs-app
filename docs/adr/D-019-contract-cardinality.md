# D-019 — Cardinality Client↔Contract: 1:0..1; contract_snapshot в акті; договір обов'язковий перед генерацією акту

**Дата:** 2026-05-17

**Рішення:**

1. **Cardinality Client ↔ Contract = 1 : 0..1.** Клієнт може існувати без договору тимчасово (сценарій з D-011 — швидке створення локального клієнта з невідомого платежу), але це проміжний стан.

2. **Договір обов'язковий перед генерацією акту для всіх service_type** (access, sms, other). Зразок акту (`research/act-sample/act-2026-04_200.pdf`) посилається на договір у юридичному тексті ("по договору №X від dd.mm.yyyy"), без цих даних стандартна форма акту не складається. Навіть нестандартні платежі (Авалон, Урбан) реально мають договір з `is_standard=false`.

3. **Перевірка в логіці класифікації** (крок 5): якщо в клієнта немає договору — додається до missing-списку причини `client_incomplete`. Адмін у адмінці бачить `missing=["contract"]` і знає, що треба внести договір.

4. **Snapshot номера і дати договору в акт.** Нове поле `Act.contract_snapshot` (JSON):
   ```json
   {
     "number": "557259",
     "signed_date": "2024-07-01"
   }
   ```
   Зберігається в момент генерації акту разом з `client_snapshot`. Не оновлюється при подальших змінах договору. Гарантує юридичну цілісність історичних актів.

**Альтернативи:**

- 1:1 строгий (як зараз). Відкинуто: блокує сценарій D-011, де адмін створює клієнта з невідомого платежу до того, як зібрав інфу про договір.
- 1:N (клієнт може мати кілька договорів — старі і новий). Відкинуто: бізнес-модель не вимагає історії договорів, а snapshot в актах гарантує юридичну цілісність незалежно.
- FK на contract в акті замість snapshot + immutable Contract. Відкинуто: суперечить snapshot-патерну D-010, ускладнює CRUD договору (заборона редагування при наявності актів).
- Окремий reason `contract_missing` замість `client_incomplete`. Відкинуто: contract з точки зору адміна — це частина "повного клієнта", а не окрема сутність. Уніфікований UX через один reason з missing-списком (D-017).
- Опціональний договір для service_type=other. Відкинуто: навіть нестандартні платежі (Авалон, Урбан) реально мають договір; уніфіковане правило простіше.

**Обґрунтування:**

- Зразок акту явно посилається на договір — це юридична вимога стандартної форми.
- 1:0..1 дозволяє "lazy" створення клієнтів з невідомих платежів без втрати валідності моделі.
- Snapshot consistent з D-004 і D-010 (історична незмінність актів).

**Наслідки:**

- В моделі Contract: cardinality зміна (1:1 → 1:0..1) — фіксується в описі сутності.
- В моделі Act: нове поле `contract_snapshot` (JSON, not null).
- В логіці класифікації крок 5: додається перевірка `IF client.contract IS NULL: missing.append("contract")`.
- В розділі snapshot-полів Act: `contract_snapshot` додається до списку.
- Документується паралельний приклад "contract_snapshot JSON структура" поряд з client_snapshot.
- В CRUD адмінці (PRD розділ 8.4 — Клієнти): при створенні нового клієнта договір не обов'язковий, але видно warning "Без договору акти не генеруються — додайте договір на вкладці 'Договори'".
- Версія `domain_model.md` → 0.6 (мінор: новий snapshot, нова перевірка, нова cardinality).

**Переглядає:** D-011 (там припускалось, що клієнт без moeosbb_user_id повністю функціональний; тепер уточнюється — без договору акти заблоковані; це уточнення, не перегляд суті).

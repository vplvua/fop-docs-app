# D-018 — Видалити Tariff.is_default; інваріант "сітка завжди має catch-all правило"

**Дата:** 2026-05-17

**Рішення:**

1. Поле `Tariff.is_default` (bool) видаляється з моделі.
2. Вводиться інваріант: **тарифна сітка завжди містить рівно одне catch-all правило** з `apartments_max IS NULL`. Інші правила можуть бути ranged (з конкретним `apartments_max`).
3. Валідація інваріанту виконується на рівні CRUD адмінки (розділ 8.7 PRD): не дозволяти видалення/збереження стану, де catch-all правила немає або їх кілька.
4. Логіка `resolve_access_price` спрощується: видаляється fallback гілка ("Fallback — default tariff"), бо `tariff_lookup` гарантовано знайде правило.
5. **Пріоритет при множинному матчингу — найбільш специфічне правило**. Порядок сортування:
   - ranged правила (з конкретним `apartments_max`) перед catch-all (`apartments_max IS NULL`);
   - серед однакового типу — менший діапазон (`apartments_max - apartments_min`) вище;
   - серед однакового діапазону — найновіше `effective_from` вище (версіонування правил у часі).

**Альтернативи:**

- Лишити `is_default` як редundант з тим самим змістом. Відкинуто: дублює семантику catch-all правила, створює можливість конфліктних станів (два `is_default=true` або жодного).
- Дозволити сітку без catch-all + hardcoded fallback константа в коді. Відкинуто: суперечить D-013 (хотіли уникнути hardcoded цін).
- Замість catch-all правила — окрема таблиця `DefaultTariff`. Відкинуто як надмірне відокремлення тієї самої сутності.

**Обґрунтування:**

- Після D-017 (`apartments_count` обов'язковий для access без override) первинна функція `is_default` (fallback при NULL apartments_count) стала непотрібною.
- Сценарій "apartments_count є, але немає правила, що його покриває" обробляється elegantно через інваріант catch-all — без додаткових полів і без окремої гілки в коді.
- Семантика "правило для всіх" виражається структурою (`apartments_max IS NULL`), а не окремим прапором — це data-driven, а не state-driven.

**Наслідки:**

- Видалення поля `Tariff.is_default` з моделі.
- Стартова конфігурація: `{apartments_min=0, apartments_max=NULL, price=200}` (без `is_default`).
- Псевдокод `resolve_access_price` спрощується: видаляється блок "Fallback — default tariff"; описується порядок сортування при множинному матчингу.
- PRD розділ 8.7 (Налаштування) отримує вимогу: валідація catch-all правила при збереженні сітки.
- В UI адмінки сітки тарифів — візуально розрізнити catch-all правило (наприклад, рядок з міткою "Для всіх інших" або відображення `apartments_max = ∞`).
- Версія `domain_model.md` → 0.5 (мінор, бо видалене поле і змінена логіка резолву).

**Переглядає:** D-013 (там Tariff згадується як сутність з `is_default` — фактична зміна структури сутності, не суті рішення).

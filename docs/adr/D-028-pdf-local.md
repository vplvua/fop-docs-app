# D-028 — PDF актів генерується локально через headless browser; в Дубідок передається готовий файл

**Дата:** 2026-05-18

**Рішення:**

1. PDF акту **рендериться локально** в нашій системі. У Дубідок передається готовий PDF через `POST /api/v1/documents` (поле `file` як base64).
2. Шаблон акту — **HTML + Tailwind**, побудований як React-компонент (htmldocs DX або еквівалент). Це дає WYSIWYG-preview в адмінці, легку правку юридичних формулювань, спільну стилістику з UI 8.x.
3. Рендер виконується **на нашому сервері** через headless browser (Puppeteer / Playwright / Chromium у Docker; або self-host `@htmldocs/render`). **Хмарне API htmldocs.com (`POST /api/generate`) не використовується** — дані клієнтів (ЄДРПОУ, реквізити, суми) не виходять за периметр сервера.
4. Конкретний рантайм (Puppeteer vs Playwright vs альтернатива) — імплементаційна деталь розділу 10.1; принципова вимога — рендер у нашій інфраструктурі.
5. Преміум-поля `POST /documents`: передаємо `number` (з `Act.number`), `date` (з `Act.act_date`), `amount` (з `Act.total_amount`), `participants[]` (email + ЄДРПОУ клієнта зі `client_snapshot`) для авто-відправки підписаного акту клієнту після підпису власником.

**Альтернативи:**

- **Хмарне API htmldocs.com.** Відкинуто: client PII (ЄДРПОУ, IBAN, адреси, email) виходять у third-party. Операційна залежність від їхнього SLA. Платні виклики на кожен акт. Бізнес-ризик не вартий зекономлених ~50 рядків коду інфраструктури.
- **Передавати структуровані дані в Дубідок, рендер у Дубідок.** Технічно неможливо: API Дубідок (`POST /api/v1/documents`) приймає тільки готовий файл (`file` base64). Шаблони Дубідок (`/api/v1/templates`) — це збережені файли, не data-binding.
- **DOCX-шаблон + LibreOffice конвертація.** Відкинуто: додаткова залежність (LibreOffice binary), DOCX→PDF дає менш передбачуваний рендер ніж HTML→PDF, складніше підтримувати шаблон у синхронізації з brand-стилем web-адмінки.
- **Програмний PDF (ReportLab, pdf-lib).** Відкинуто: кожна правка юридичного тексту = код-реліз. Не підходить для документів, що періодично коригуються (формулювання, реквізити ФОПа).

**Обґрунтування:**

- API Дубідок не дає опції "структуровані дані" — рендер у нас обов'язковий.
- HTML+Tailwind+React як шаблон — найкращий DX серед локальних опцій: візуальний preview, common stack з адмінкою, легка правка тексту.
- Self-hosted рендер закриває privacy і operational concerns без втрати DX (htmldocs OSS пакети `htmldocs`, `@htmldocs/react`, `@htmldocs/render` дають DX без cloud lock-in).
- Преміум Дубідок підтверджений → можемо повноцінно використати `participants` для авто-розсилки клієнту після підпису власником. Це закриває крайову точку процесу без додаткових інтеграцій з email.

**Наслідки:**

- В PRD 6.3 опис шаблонів формулюється в термінах HTML+React+Tailwind.
- В PRD 7.2 (Дубідок) фіксується спосіб передачі: file (base64) + Premium-поля number/date/amount + participants.
- В PRD 10.1 (стек) додається залежність — headless browser у production-середовищі (Chromium у Docker або еквівалент).
- В PRD 10.3 (Безпека) — додатковий пункт: рендер відбувається в нашому середовищі; жодні client PII не передаються в третісторонні рендер-сервіси.
- В моделі `Act` поле `pdf_file_url` тепер — це посилання на наш storage (S3 або local FS, рішення на етапі 10.1), не на htmldocs.com.
- Модель `Act` без змін.

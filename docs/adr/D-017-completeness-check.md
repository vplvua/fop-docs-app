# D-017 — Перевірка повноти даних клієнта: умовна для apartments_count, до резолву ціни

**Дата:** 2026-05-17

**Рішення:** Логіка перевірки `client_incomplete` змінюється:

1. **Перевірка переноситься ДО резолву ціни** (з кінця класифікації на середину, перед визначенням unit_price).

2. **Умова обов'язковості полів стає диференційованою:**
   - `email` — обов'язковий завжди (Дубідок надсилає підписаний акт автоматично, без email клієнт не отримає документ).
   - `address` — обов'язковий завжди (рендериться як реквізит у PDF акту).
   - `apartments_count` — обов'язковий ТІЛЬКИ якщо `service_type == access` І `access_price_override IS NULL`. Не потрібен для `sms`, `other`, або access з override.

3. **Reason `client_incomplete` отримує деталізацію** (список конкретних missing-полів) — обчислюється UI на льоту з поточного стану клієнта, **не зберігається** в `Payment`. Якщо з часом стане потрібна історія "що було missing на момент класифікації" — додамо поле окремим рішенням.

**Новий порядок кроків логіки класифікації:**

| #   | Крок                                          | Зміна                                             |
| --- | --------------------------------------------- | ------------------------------------------------- |
| 1   | Парсинг номера договору                       | без змін                                          |
| 2   | Матчинг клієнта                               | без змін                                          |
| 3   | Перевірка auto_act_disabled → awaiting_review | без змін (D-014)                                  |
| 4   | Визначення типу послуги (access/sms)          | без змін                                          |
| 5   | **Перевірка повноти даних** (новий)           | переміщений з кроку 7, з диференційованою логікою |
| 6   | Резолв ціни                                   | був крок 5                                        |
| 7   | Визначення quantity                           | був крок 6                                        |
| 8   | Генерація акту                                | без змін                                          |

Старий крок 7 видаляється (його логіка інтегрована в крок 5).

**Альтернативи:**

- Лишити `apartments_count` як обов'язковий завжди. Відкинуто: для sms/other він ніде не використовується, для access з override — також. Вимагати фейкове значення → шум у даних, плутанина для адміна, особливо для клієнтів-ФОП-управителів, де "квартири" нерелевантні.
- Лишити перевірку після резолву ціни. Відкинуто: при відсутності apartments_count + відсутності override крок резолву використовує fallback на default tariff (200 грн), що дає некоректний `quantity` у UI — адмін бачить фейковий розрахунок замість чесного "не можемо порахувати".
- Прибрати обов'язковість `email` (адмін може хотіти контролювати, чи надсилати). Відкинуто: відсутність email = клієнт не отримає документ, це бізнес-критично.
- Додати поле `Payment.classification_details` (JSON) для зберігання missing-полів. Відкинуто на цьому етапі: деталі завжди можна обчислити з поточного стану клієнта в UI. Якщо потрібна історія — додамо потім.

**Обґрунтування:**

- Зменшує false-positives у черзі: не вимагаємо даних, які реально не потрібні.
- Покращує UX адмінки: коли платіж в `in_queue(client_incomplete)`, видно конкретно, ЧОГО не вистачає (`missing: ["apartments_count"]` vs `missing: ["email", "address"]`).
- Семантично коректний порядок: спочатку перевіряємо, чи можемо порахувати → потім рахуємо. Не навпаки.

**Наслідки:**

- Псевдокод логіки класифікації перебудовується (порядок кроків + умова на apartments_count).
- Версія `domain_model.md` → 0.4 (мінорна, бо змінюється логіка).
- `is_default` на Tariff лишається релевантним (fallback при відсутності матчингу в сітці зберігається — це окрема ситуація від відсутності apartments_count). Окремо обговорюється в пункті 5 нашого списку.

**Переглядає:** початкову версію кроку 7 в логіці класифікації.

# D-025 — ON DELETE: RESTRICT для Client/Contract; архівування через auto_act_disabled

**Дата:** 2026-05-18

**Рішення:**

1. FK з `Payment.client_id` і `Act.client_id` на `Client` мають `ON DELETE RESTRICT`. БД блокує видалення клієнта, якщо існує хоча б один платіж або акт.
2. FK з `Contract.client_id` на `Client` — `ON DELETE RESTRICT` (договір не видаляється каскадно; договір — юридичний документ).
3. FK з `Payment.act_id` на `Act` — `ON DELETE SET NULL` (акт можна видалити з БД у виключних випадках; платіж має лишитись і може бути повторно класифікований).
4. FK з `Act.payment_id` на `Payment` — `ON DELETE RESTRICT` (платіж не видаляється, якщо існує акт).
5. Для побутового "приховування" клієнта, з яким більше не працюємо, використовується **прапор `auto_act_disabled=true` + UI-фільтр "Активні / Архів"** на сторінці клієнтів (PRD 8.4). Окремого поля `deleted_at` не вводимо.

**Альтернативи:**

- **Soft delete** (поле `deleted_at` на Client/Contract). Відкинуто: для single-admin це надлишкова інфраструктура (треба пам'ятати фільтр у кожному запиті, є ризик показати "видаленого" клієнта в одному з місць). `auto_act_disabled` уже виконує функцію "виключити з робочого потоку".
- **ON DELETE CASCADE.** Відкинуто: каскадне видалення актів — втрата юридичних документів.
- **Окремий прапор `is_archived` поряд з `auto_act_disabled`.** Відкинуто: дублює семантику. Якщо клієнта архівують — авто-генерація актів все одно має бути вимкнена. Один прапор простіший.

**Обґрунтування:**

- Акти — юридичні документи з обов'язком зберігання. RESTRICT на рівні БД гарантує, що випадкова SQL-команда або баг не видалить історію.
- Snapshot-поля (D-010, D-019) роблять Act самодостатнім: видалення клієнта не псує рендер історичного PDF, але втрачає посилання для аналітики (фільтр по клієнту, історія актів клієнта). RESTRICT зберігає цей зв'язок.
- `auto_act_disabled` уже існує (D-003) і виконує функцію "виключити з авто-обробки". Перевикористання прапора простіше за нову сутність.
- Випадок з `Payment.act_id ON DELETE SET NULL`: якщо в Дубідок акт видалили помилково — обнуляємо посилання на Payment, потім створюємо новий Act з повторного запуску класифікації. Узгоджується з 9.9.

**Наслідки:**

- В `domain_model.md` під кожним FK прописується ON DELETE поведінка.
- В UI 8.4 (Клієнти) фільтр "Активні / Архів" базується на `auto_act_disabled`.
- В PRD 9.5 (зміна реквізитів) — snapshot гарантує незмінність історичних актів; видаляти/підмінювати клієнта не потрібно.
- В PRD 9.9 (видалення акту в Дубідок) — модель готова через `Payment.act_id` SET NULL.
- Версія `domain_model.md` → 0.7.2 (patch — додавання поведінки FK без зміни полів).

# D-041 — EDRPOU-first матчинг клієнта (переглядає D-009)

**Дата:** 2026-05-29

**Рішення:** Первинним фактором ідентифікації клієнта стає **ЄДРПОУ платника** (`payment.payer_legal_id`), а не номер договору. Номер договору використовується лише як вторинний розрізнювач серед клієнтів з тим самим ЄДРПОУ і ніколи не перевизначає компанію.

Логіка для не-транзитних платежів:

1. `candidates` = клієнти з `legal_id == payer_legal_id`; `activeCandidates` = ті з них, у кого `auto_act_disabled = false`.
2. `candidates` порожній → `in_queue(no_match)`.
3. Рівно 1 активний → matched (номер договору, навіть хибний, ігнорується).
4. > 1 активний → розрізнення по договору: рівно 1 збіг → matched; інакше → `awaiting_review(multiple_clients_same_edrpou)` зі списком активних кандидатів для ручного вибору. Якщо при цьому в призначенні >1 різних номерів договору — `in_queue(multiple_contracts)` (D-027 застосовується лише коли договір реально розрізнює).
5. 0 активних, але є архівовані → `awaiting_review(auto_act_disabled)`; `client_id` = архівований, якщо він один (зберігає D-014/D-003 для exclusion-list).

Транзит (D-008) зберігається: матч по договору серед активних клієнтів, без перевірки ЄДРПОУ.

Ручна привʼязка платежу до клієнта дозволена лише в межах ЄДРПОУ платника (виняток — транзит, де клієнт визначається договором).

**Альтернативи:**

- **Лишити контракт-first (D-009).** Відкинуто: номер договору вводить платник і регулярно помиляється; хибний номер тихо привʼязував платіж до неправильного запису того ж ЄДРПОУ.
- **Повністю виключити архівованих з матчингу.** Відкинуто: зламало б exclusion-list (Урбан/Авалон → `no_match` замість `awaiting_review`).
- **«Завжди селектор при >1 кандидаті».** Відкинуто: незнищенні дублікати-сміття від синку Моє ОСББ змушували б клікати селектор на кожному платежі. «Перевага єдиному активному» авто-вирішує цей кейс.

**Обґрунтування:**

- ЄДРПОУ — власний реквізит платника, надійний; номер договору — ні.
- Реальний кейс «РЕСПЕКТ»: один активний клієнт + 5 архівованих дублікатів (невидаляльні — синк відновлює, договір обовʼязковий). Платник вказав чужий номер → платіж тихо сідав на архівований дублікат. EDRPOU-first вирішує це автоматично.
- Неоднозначність (кілька активних на один ЄДРПОУ) стає **явною** через `multiple_clients_same_edrpou` замість тихого вибору.

**Наслідки:**

- `classification_reason` зберігається як `text` (не Postgres ENUM) — нове значення `multiple_clients_same_edrpou` **не потребує міграції БД**. Список активних кандидатів кодується в reason як `multiple_clients_same_edrpou:<id1>,<id2>`.
- `lib/classification/match-client.ts` переписано; крок `multiple_contracts` перенесено з `classify.ts` всередину матчингу (спрацьовує лише як розрізнювач).
- Нова server action `linkPaymentClientAction` + інваріант `lib/classification/link-validation.ts`.
- UI платежу: попередження + селектор активних кандидатів (`classification-panel.tsx`).
- Причина `ambiguous_client` більше не виникає (ЄДРПОУ-first ніколи не матчить через ЄДРПОУ-конфлікт); значення лишається в коді для історичних платежів.
- Після деплою — повторна класифікація наявних `in_queue`/`awaiting_review`.

**Переглядає:** D-009 (двофакторний матчинг: контракт-first → EDRPOU-first). Уточнює межі D-008 (транзит), D-014 (awaiting_review/auto_act_disabled), D-027 (multiple_contracts як розрізнювач), D-003 (exclusion-list).

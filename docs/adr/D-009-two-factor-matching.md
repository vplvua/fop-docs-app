# D-009 — Двофакторний матчинг клієнта: договір + ЄДРПОУ

**Дата:** 2026-05-16

**Рішення:** При матчингу клієнта по номеру договору додатково перевіряємо, що ЄДРПОУ платника збігається з ЄДРПОУ знайденого клієнта. Якщо не збігається — платіж попадає в чергу розбору з `reason=ambiguous_client`.

**Альтернативи:**

- Довіряти тільки номеру договору (як було в попередній моделі).

**Обґрунтування:** Аналіз виписки виявив 8 випадків, коли один і той самий номер договору зустрічається в платежах від різних ЄДРПОУ. Можливі причини: помилка платника (вказав чужий номер), реальний дубль номерів, помилка розпізнавання. У всіх випадках це повинно тригерити ручну перевірку, а не автокласифікацію.

**Наслідки:**

- Окрема enum-причина в `Payment.classification_reason`: `ambiguous_client`.
- Логіка класифікації обробляє три гілки: збіг → ок, неузгодженість → черга, не знайдено → fallback на ЄДРПОУ.
- Виняток: транзитний рахунок ПриватБанку, де ЄДРПОУ платника — це ЄДРПОУ банку, не реального платника. Для транзиту перевірка ЄДРПОУ пропускається.

# D-014 — Payment.status: 5 станів, awaiting_review окремо від in_queue

**Дата:** 2026-05-17

**Рішення:** Enum `Payment.status` розширюється з 4 до 5 значень:

- `received` — щойно з ПриватБанку, не класифіковано
- `classified` — успішно класифіковано, акт згенеровано автоматично
- `awaiting_review` — клієнт ідентифікований, але авто-генерація акту заблокована бізнес-правилом (зараз єдиний випадок — `client.auto_act_disabled = true`). Потрібен ручний апрув/корекція перед генерацією акту.
- `in_queue` — автокласифікація неможлива через проблему з даними (немає матчингу, конфлікт ЄДРПОУ, невідповідність суми, неповнота даних клієнта тощо). Потрібне ручне втручання для з'ясування ситуації.
- `skipped` — пропущено вручну, акт не потрібен (термінальний стан).

**Альтернативи:**

- Лишити 4 стани (`received / classified / in_queue / skipped`) і розрізняти "потрібен ручний штрих" vs "проблема в даних" тільки через `classification_reason`.
- Замінити `skipped` на `processed` (як у PRD_skeleton.md) — відкинуто як менш точне.

**Обґрунтування:**

- Семантично "клієнт у списку виключень (Урбан/Авалон), бо процес нестандартний" і "ми не змогли зрозуміти, хто платить" — різні стани, що вимагають різного UX в адмінці. Перший — рутина, другий — інцидент.
- `in_queue` як єдиний "не-щасливий" стан перевантажений: фільтр "Черга розбору" в UI або змішує два сценарії, або вимагає допфільтрації по `classification_reason`.
- Окремий стан `awaiting_review` дозволяє в UI зробити окрему вкладку "На апрув" з готовим розрахунком, відмінну від "Проблеми класифікації".

**Наслідки:**

- `Payment.status` enum: 5 значень.
- `classification_reason` для `awaiting_review` має поки що одне значення: `auto_act_disabled`. Для `in_queue`: решта (no_match, ambiguous_client, amount_mismatch, sms_quantity_mismatch, unknown_service_type, client_incomplete).
- Логіка класифікації крок 3 (перевірка `auto_act_disabled`) повертає `awaiting_review`, не `in_queue`.
- State machine Payment отримує два альтернативних non-success переходи: `received → awaiting_review` і `received → in_queue`. Обидва можуть вести в `classified` або `skipped`.
- В адмінці (PRD розділ 8.3 "Черга розбору") з'являється поділ на дві вкладки/секції.
- PRD_skeleton.md розділ 3.3 оновлюється.

**Переглядає:** D-003 (там `auto_act_disabled` згадується як reason для in_queue — тепер це reason для awaiting_review).

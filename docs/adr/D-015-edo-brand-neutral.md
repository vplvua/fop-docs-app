# D-015 — Бренд-нейтральні поля EDO замість vchasno\_\*

**Дата:** 2026-05-17

**Рішення:** Поля моделі, що стосуються електронного документообігу, перейменовуються з префіксу `vchasno_` на бренд-нейтральний `edo_` (electronic document operator). Виняток — credentials, що зберігають префікс конкретного провайдера.

| Було                           | Стало                      | Чому саме так                        |
| ------------------------------ | -------------------------- | ------------------------------------ |
| `Contract.vchasno_doc_id`      | `Contract.edo_doc_id`      | Абстракція "ID документа в EDO"      |
| `Act.vchasno_doc_id`           | `Act.edo_doc_id`           | Те саме                              |
| `Act.vchasno_status`           | `Act.edo_status`           | Статус в EDO в загальному            |
| `Act.status = sent_to_vchasno` | `Act.status = sent_to_edo` | Транзит акту в EDO                   |
| `Settings.vchasno_token`       | `Settings.dubidoc_token`   | Credential специфічний до провайдера |

**Альтернативи:**

- `dubidoc_*` всюди (vendor-coupled). Відкинуто: можлива майбутня інтеграція з Vchasno створить rename pain і плутанину типу "цей `dubidoc_doc_id` точно від Дубідок чи від іншого?".
- `edo_*` всюди включно з токеном. Відкинуто: один універсальний `edo_token` не працює, бо credentials провайдер-специфічні (формат, endpoint, headers).
- Повне multi-provider моделювання (поле `edo_provider` на акті, окрема таблиця провайдерів). Відкинуто як speculative для MVP — зараз 100% акти йдуть в один провайдер.

**Обґрунтування:**

- Vchasno (vchasno.ua, від EVO) і Дубідок (dubidoc.com.ua, від ПриватБанку) — різні сервіси. Vchasno-як-помилка в існуючих іменах полів — артефакт з історії, не свідомий вибір.
- На рівні моделі `edo_doc_id` описує абстракцію "ID документа в EDO системі". Це не передбачає multi-provider, але не блокує його.
- Існуючий код для Дубідок є в `~/Projects/zbory_v2` (webhook'и під шляхом `/api/webhooks/dubidoc/`). Реюз працює незалежно від назв наших полів.

**Наслідки:**

- 4 перейменування на `edo_*` (Contract, Act × 3), 1 rename на `dubidoc_token` (Settings).
- В людських текстах (PRD, decisions log, doc-коментарі) поточний провайдер називається "Дубідок", абстракція — "EDO".
- Модель НЕ підтримує одночасну роботу з кількома EDO. Якщо такий use case виникне — окреме рішення D-XXX про:
  - або додавання `edo_provider` поля на акт/контракт,
  - або вибір "primary" EDO в глобальних налаштуваннях.
- Запис у `CHANGELOG.md`, версія `domain_model.md` → 0.2.1 (косметичний patch).

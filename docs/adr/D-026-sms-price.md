# D-026 — Версіонування sms_unit_price у окремій таблиці SmsPrice

**Дата:** 2026-05-18

**Рішення:**

1. Вводиться нова сутність `SmsPrice` симетрично до `Tariff`:
   - `id` (uuid)
   - `price` (numeric(10,2)) — ціна за 1 СМС на дату дії
   - `effective_from` (date) — дата початку дії правила
   - `created_at` (timestamptz)
2. Резолв ціни СМС для класифікації платежу:
   ```
   resolve_sms_price(payment_date):
       SELECT price FROM sms_prices
       WHERE effective_from <= payment_date
       ORDER BY effective_from DESC
       LIMIT 1
   ```
3. З `Settings` видаляються ключі `sms_unit_price` і `sms_unit_price_effective_from` — їх роль перебирає таблиця `SmsPrice`.
4. Стартова конфігурація: один рядок `{effective_from='2024-01-01', price=1.40}`.
5. `Settings.sms_min_quantity` (100 шт) — лишається в Settings, бо це бізнес-правило мінімального замовлення, не цінова історія. Якщо колись доведеться версіонувати — окреме рішення.
6. CRUD `SmsPrice` доступний з UI 8.7 (Налаштування) — додати нову ціну, відредагувати майбутню, видалити майбутню (історичні ціни не редагуються після того, як на них згенерувались акти).

**Альтернативи:**

- **jsonb-масив `Settings.sms_price_history`.** Відкинуто: гірша валідація (нема FK-механіки, треба обходити jsonb у резолві), асиметрія з Tariff, де версіонування вже зроблено таблицею.
- **Одне поточне значення без історії.** Відкинуто: відкладений платіж (PRD 9.10), який прийшов до зміни ціни, але класифікується після — рахуватиметься за новою ціною. Юридично некоректно: ціна послуги визначається на дату оплати, не на дату обробки в нашій системі.
- **Логувати зміни в окрему audit-таблицю, але резолв брати з поточного значення Settings.** Відкинуто: audit-лог не той самий механізм, що історичний резолв. Резолв за датою потребує саме taблиці з `effective_from`.

**Обґрунтування:**

- Симетрія з `Tariff`: для access-послуги ми вже маємо історизацію цін через `effective_from` (D-013, D-018). Для sms-послуги логіка тотожна, тому структура має бути такою ж.
- Snapshot в `Act.unit_price` (D-010) гарантує незмінність історичних актів, але не покриває сценарій 9.10: платіж від 1 квітня обробили 5 травня; якщо ціна змінилась 1 травня, snapshot повинен зафіксувати ціну на 1 квітня, а не на 5 травня. Без `SmsPrice.effective_from` цього не зробити.
- PostgreSQL обраний у D-024 — таблиця з 2-3 рядків і простим запитом — це нормальна вартість.

**Наслідки:**

- В моделі: нова сутність `SmsPrice` поряд з `Tariff`.
- В `Settings` видаляються `sms_unit_price`, `sms_unit_price_effective_from`.
- В псевдокоді класифікації крок 6 (резолв ціни для sms) посилається на `resolve_sms_price(payment_date)` замість `settings.sms_unit_price_at(...)`.
- В PRD 5.5 опис уточнюється: SMS-ціна тримається в окремій сутності, не в Settings.
- В PRD 8.7 (Налаштування) додається CRUD для SmsPrice.
- В PRD 9.10 — резолв за датою платежу через таблицю, не за поточним Settings.
- Версія `domain_model.md` → 0.8 (мінор: нова сутність + видалення двох ключів Settings).

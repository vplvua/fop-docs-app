## 1. Контекст і мета

### 1.1 Бізнес-контекст

**Автор** — ФОП на 3 групі єдиного податку, водночас власник і єдиний оператор SaaS-продукту **"Моє ОСББ"**. Цільова аудиторія — об'єднання співвласників багатоквартирних будинків (ОСББ), житлово-будівельні кооперативи (ЖБК), ТОВ та ФОП-управителі, які адмініструють житлові будинки. Зараз база — ~185 активних контрагентів (за останні 5 міс. виписки); цільовий масштаб системи — ~300 клієнтів.

**Платежі і послуги:**

- Основна послуга — **доступ до сервісу** "Моє ОСББ". Стартова ціна 200 грн/міс або 2000 грн/рік. На дату підготовки PRD у системі одне catch-all правило тарифу (200 грн), готова інфраструктура диференціації за `apartments_count` (D-013, D-018).
- Додаткова послуга — **СМС-розсилки** для мешканців (1.40 грн/шт, мінімум 100 шт за замовлення). Ціна версіонована в часі через окрему сутність `SmsPrice` (D-026).
- Гроші приходять на розрахунковий рахунок ПриватБанку (`UA963052990000026005031009982`).
- Виписка за останні 5 місяців: **472 платежі, 431 524,94 грн**. У середньому ~94 платежі/місяць; пікові місяці — 117 платежів. Цільовий розрахунковий профіль — до 500 платежів/місяць після зростання бази.

**Поточний процес (станом на дату підготовки PRD):**

1. Власник періодично (раз на тиждень / два тижні) вручну переглядає виписку у Приват24.
2. Для кожного платежу вручну ідентифікує клієнта за призначенням платежу або ЄДРПОУ.
3. Заходить у Дубідок (EDO-провайдер від ПриватБанку), створює документ "Акт виконаних робіт" з шаблону, вписує реквізити клієнта, номер і дату договору, суму, кількість послуги.
4. Підписує електронним підписом у Дубідок (фіз.особа підписує вручну — D-001).
5. Дубідок автоматично надсилає підписаний акт на email клієнта (на Premium-тарифі — D-029).

При обсязі ~100 актів/місяць цей процес займає від 8 до 15 годин на місяць (за оцінкою власника), при цьому містить регулярні класи помилок: неправильно зчитані номери договорів, плутанина двох клієнтів з однаковим іменем, втрата актів через відволікання адміна посеред пакета.

**Біль і мотивація:**

- Витрати часу на рутинну операцію, яку реально автоматизувати на 90%+.
- Помилки реквізитів — повторне виставлення акту, неприємні розмови з клієнтами.
- Сезонність (грудень+січень — пік платежів через річні оплати), яка створює пакети по 3-4 години роботи поспіль.
- Психологічна вартість "тримати в голові", чи всі акти за минулий місяць виставлені.

**Юридичний контекст:**

- ФОП на 3 групі ЄП виставляє акт **за фактом оплати**, не за фактом надання послуги (D-007). Один платіж = один акт; дата акту = останній календарний день місяця платежу (D-006).
- Електронний підпис у Дубідок не дозволяє підписувати документ до настання його дати — тому акт за платіж 5 квітня матиме дату 30.04 і буде підписаний 1 травня або пізніше.

### 1.2 Мета документа

**Цей PRD описує:**

- Систему автоматичної генерації актів виконаних робіт з мінімальною адмін-частиною: polling ПриватБанку, класифікація платежів, генерація PDF, відправка в Дубідок, віртуальне відстеження статусів.
- Доменну модель, логіку класифікації, цінову політику, інтеграції, адмін-інтерфейс і edge cases — все, що потрібне команді з 1 розробника, щоб збудувати і запустити Phase 0 (розділ 11).

**Цей PRD НЕ описує:**

- Продукт "Моє ОСББ" як цілісну SaaS-платформу — це окремий проект, з яким наша система інтегрується **read-only** (D-001, D-004).
- Облік оплат у "Моє ОСББ" (продовження дат блокування акаунту після оплати, поповнення СМС-балансу) — це ручний процес автора, який залишається ручним.
- Підпис актів — це ручна дія власника в Дубідок UI (D-001).
- Звітність ФОП, податковий облік, обчислення ЄП-внесків — поза скоупом.
- Робота з історичними платежами до дати запуску системи (D-001) — стартуємо з "чистого аркуша".

**Цей PRD узгоджений з:**

- `domain-model.md` (поточна версія 0.9.1) як канонічним джерелом моделі.
- `adr/` D-001…D-035 як архітектурно-продуктовим логом рішень.
- `domain-model-changelog.md` як технічним журналом правок моделі.

**Що НЕ є метрикою:**

- Швидкість підпису акту (це поза нашим контролем — це ручна операція власника).

- Точність розпізнавання нестандартних договорів (10% складних кейсів попадають у чергу — це за дизайном, не bug).

## 2. Користувачі і сценарії

### 2.1 Користувачі системи

**Єдиний користувач — Адмін** (= автор системи = власник ФОП = власник "Моє ОСББ"). Single-admin сценарій явно зафіксовано в D-001.

**Контекст роботи адміна:**

- Заходить у систему 1-2 рази на тиждень (рутина), або частіше в дні пакетних надходжень (1-2 числа місяця, грудень/січень).
- Працює переважно з домашньої мережі.
- Має доступ до: всіх клієнтських даних (PII), фінансових сум, договорів, токенів API (через env-керування — D-020).
- Має повний контроль (CRUD) над усіма сутностями, окрім полів, що приходять із sync "Моє ОСББ" і `auto_act_disabled`/`apartments_count`, що manual-only.

**Хто НЕ є користувачем системи:**

- Клієнти ОСББ — вони отримують підписані акти на email, але не мають доступу до адмінки. Їхня взаємодія з нашою системою — нульова за дизайном.
- Бухгалтер автора (якщо є) — поза скоупом. Якщо в Phase 2 з'явиться потреба — окреме D-XXX про read-only роль.
- Розробник — це той самий автор у іншій ролі. Працює через GitHub + Vercel UI, не через адмінку.

**Авторизація** — email/password без 2FA в MVP, credentials у env (D-032). Деталі — 8.8.

### 2.2 Основні сценарії використання

Сценарії впорядковані за частотою. Кожен — це end-to-end користувацька історія, де адмін взаємодіє з системою для досягнення конкретного результату.

---

**Сценарій 1: Щотижневий огляд платежів та черги розбору** (найчастіший)

**Тригер:** адмін заходить у систему періодично (раз/тиждень), щоб переконатись, що нічого не зависло.

**Кроки:**

1. Login (8.8) — email/password.
2. Відкриває Дашборд (8.1):
   - Перевіряє банери стану зовнішніх API — все має бути "ОК".
   - Дивиться на лічильники: "Платежів у черзі: X", "На апрув: Y", "Актів очікують підпису: Z".
3. Якщо черга порожня (X+Y=0) — закриває систему. Адмін витратив 30 секунд, переконався, що система працює.
4. Якщо в черзі є платежі — переходить у "Чергу розбору" (8.3) і виконує сценарій 2 для кожного.
5. Якщо є акти `awaiting signing` довго — заходить у "Акти" (8.6), перевіряє, чи треба підписати їх у Дубідок (це ручна операція поза системою).

**Очікуваний час:** 1-5 хвилин в нормальному тижні; 15-30 хвилин у пакетні дні.

---

**Сценарій 2: Розбір нерозпізнаного платежу** (платіж у черзі з reason `no_match`, `multiple_contracts`, `ambiguous_client` або `client_incomplete`)

**Тригер:** у черзі розбору 8.3 (Tab "Проблеми класифікації") є платіж, який автокласифікація не змогла обробити.

**Кроки (залежно від reason):**

_Для `no_match`:_

1. Адмін бачить картку платежу: дата, сума, призначення, реквізити платника.
2. Робить пошук у "Клієнтах" (8.4) по ЄДРПОУ або частковій назві. Або перевіряє вручну, чи це може бути клієнт із "Моє ОСББ", який ще не в нашій БД.
   3a. Якщо знайшов — натискає "Прив'язати до існуючого клієнта", вибирає з autocomplete. Класифікація реруниться, акт згенерується.
   3b. Якщо не знайшов — натискає "Створити нового клієнта" (передзаповнена форма). Заповнює відсутні поля (apartments_count, email, договір). Зберігає. Класифікація реруниться.

_Для `multiple_contracts` (D-027):_

1. Картка показує знайдені номери договорів (radio-select).
2. Адмін вибирає правильний.
3. Реруниться класифікація.

_Для `ambiguous_client` (D-009):_

1. Картка показує знайденого клієнта по договору і ЄДРПОУ платника поруч.
2. Адмін: "Підтвердити цього клієнта" / "Створити іншого" / "Виправити в моделі" (наприклад, виправити legal_id у клієнта, якщо це банальна одруківка).

_Для `client_incomplete` (D-017, D-019, D-022):_

1. Картка показує список missing-полів з посиланнями на конкретні поля клієнта/договору.
2. Адмін переходить у картку клієнта (8.4), заповнює відсутнє (email, address, apartments_count, договір).
3. Повертається у чергу, реруниться класифікація.

**Очікуваний час:** 30 секунд - 2 хвилини на платіж.

---

**Сценарій 3: Створення нового клієнта при платежі від невідомого контрагента**

**Тригер:** платіж з reason `no_match`, ЄДРПОУ платника не знайдено в БД.

**Кроки:**

1. У картці платежу 8.2 адмін натискає "Створити нового клієнта".
2. Відкривається форма /clients/new з передзаповненими `name`, `legal_id`, `bank_account` з `payer_*`.
3. Адмін доповнює: `email`, `address`, `apartments_count` (тільки якщо access без override), `moeosbb_user_id` (якщо клієнт уже в "Моє ОСББ"), бажано — `access_price_override` (якщо ціна не стандартна), `auto_act_disabled` (якщо клієнт нестандартний).
4. Зберігає клієнта.
5. Адмін бачить warning "Без договору акти не генеруються" (D-019).
6. У вкладці "Договір" клієнта створює договір: number (зазвичай = `moeosbb_user_id`), signed_date, is_standard, file_url (опц.).
7. Повертається до платежу, реруниться класифікація. Якщо все ОК — акт згенерується автоматично.

**Очікуваний час:** 2-5 хвилин (більшість часу — заповнення полів клієнта і договору).

---

**Сценарій 4: Зміна тарифу для конкретного клієнта**

**Тригер:** один клієнт домовився про знижену/підвищену ціну доступу (індивідуальна угода).

**Кроки:**

1. Адмін заходить у картку клієнта 8.4.
2. У формі редагує `access_price_override` — встановлює конкретне значення (наприклад, 150 грн замість 200).
3. Зберігає.
4. Майбутні платежі від цього клієнта будуть класифіковані з новою ціною. Snapshot захищає історичні акти від змін (D-010, 5.5).

**Що сценарій НЕ включає:**

- Перевипуск історичних актів — це окремий ручний flow в 8.6 (9.5).
- Звітування клієнту про зміну ціни — комунікація поза системою.

**Очікуваний час:** 30 секунд.

---

**Сценарій 5: Зміна тарифної сітки**

**Тригер:** автор приймає бізнес-рішення про підвищення/зниження базової ціни доступу, або про введення диференціації за `apartments_count`.

**Кроки:**

1. Адмін заходить у `/settings/tariffs` (8.7).
2. Натискає "Додати правило".
3. Вписує:
   - `apartments_min`, `apartments_max` (або порожнє = catch-all)
   - `price`
   - `effective_from` — дата, з якої правило діє
4. Зберігає. Валідація перевіряє інваріант "сітка має рівно одне catch-all" (D-018).
5. Майбутні платежі (з payment_date >= effective_from) будуть класифіковані з новою ціною. Платежі з ранішою датою — за старим правилом.
6. Якщо зміна ціни робиться **ретроактивно** (нове effective_from в минулому), всі платежі, які ще не класифіковано і чий payment_date >= effective_from, отримають нову ціну при наступному запуску класифікації. Уже згенеровані акти захищені snapshot-ом (5.5).

**Очікуваний час:** 1-2 хвилини.

---

**Сценарій 6: Підпис акту в Дубідок** (поза нашою системою, але важливий контекст)

**Тригер:** новий акт у статусі `sent_to_edo`, очікує підпису власника.

**Кроки:**

1. Адмін відкриває картку акту 8.6.
2. Натискає "Перейти в Дубідок" (link на `https://my.dubidoc.com.ua/documents/{edo_doc_id}`).
3. У UI Дубідок підписує електронним підписом (КЕП / Дія / MoBank ID / Smart-ID — D-029).
4. Дубідок автоматично надсилає клієнту копію документа на email (Premium feature, D-029).
5. Через ≤6 годин (default polling — D-029) наша система виявляє `status = signed` у Дубідок і оновлює `Act.status = signed`.

**Що адмін НЕ робить у нашій системі для підпису:** жодних дій. Підпис — повністю поза скоупом системи (D-001).

## 3. Доменна модель і структура даних

**Канонічне джерело моделі — `domain-model.md`** (поточна версія 0.7.2). Цей розділ дає продуктовий огляд: які сутності існують, як вони пов'язані, які поля stateful і snapshot, і як працює локальна БД відносно "Моє ОСББ". За точними типами полів, інваріантами і псевдокодом — звертатися до `domain-model.md`.

### 3.1 Сутності і їх атрибути

| Сутність   | Призначення                                                                                                        | Origin даних                                                                                                                                          |
| ---------- | ------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Client`   | Контрагент-отримувач послуги (ОСББ, ЖБК, ТОВ або ФОП-управитель)                                                   | Sync з "Моє ОСББ" для базових реквізитів + manual для бізнес-полів (`apartments_count`, `access_price_override`, `auto_act_disabled`, `edo_provider`) |
| `Contract` | Договір з клієнтом (1 : 0..1 до Client) — D-019                                                                    | Manual; номер за замовчуванням дорівнює `moeosbb_user_id`                                                                                             |
| `Payment`  | Вхідний платіж із ПриватБанку                                                                                      | Авто з API ПриватБанку (idempotency по `bank_transaction_id`); класифікація — auto або manual                                                         |
| `Act`      | Акт виконаних робіт; один платіж = один акт (D-007)                                                                | Авто-генерація після класифікації; manual для service_type=other (D-016)                                                                              |
| `Tariff`   | Правило тарифної сітки (apartments_min..max → price, з `effective_from`)                                           | Manual в адмінці (D-013)                                                                                                                              |
| `Settings` | Глобальні key-value налаштування (regex-патерни, SMS-параметри, polling-інтервал, ЄДРПОУ транзитних, розклад sync) | Manual; credentials НЕ зберігаються тут — тільки в env (D-020)                                                                                        |

Origin-категорії для кожного поля задокументовано в `domain-model.md` (sync / manual / manual only / system).

### 3.2 Зв'язки між сутностями

Cardinality:

```
Client (1) ──── (0..1) Contract           D-019
Client (1) ──── (N)    Payment
Client (1) ──── (N)    Act
Payment (1) ─── (1)    Act                D-007 (один платіж = один акт)
```

**ON DELETE поведінка (D-025):**

| FK                               | ON DELETE  |
| -------------------------------- | ---------- |
| `Contract.client_id → Client.id` | `RESTRICT` |
| `Payment.client_id → Client.id`  | `RESTRICT` |
| `Payment.act_id → Act.id`        | `SET NULL` |
| `Act.client_id → Client.id`      | `RESTRICT` |
| `Act.payment_id → Payment.id`    | `RESTRICT` |

`RESTRICT` гарантує юридичну цілісність історії: клієнта з платежами/актами не можна видалити навіть випадковою SQL-командою. Для побутового "приховування" клієнта використовується `auto_act_disabled = true` + UI-фільтр "Архів" у розділі 8.4. `Payment.act_id` із `SET NULL` обслуговує сценарій 9.9 (акт видалили в Дубідок → платіж лишається і може бути повторно класифікований).

### 3.3 Stateful поля

**`Payment.status`** (D-014) — 5 значень з двома альтернативними non-success гілками:

```
received ──┬─→ classified  (авто, акт згенеровано)
           ├─→ awaiting_review  (ідентифікований, але бізнес-правило блокує авто-акт)
           └─→ in_queue   (автокласифікація неможлива через проблеми даних)
                          ↓ (ручний розбір)
              awaiting_review/in_queue ─→ classified | skipped
```

Стани `classified` (success) і `skipped` (свідомо пропущено) — фактично термінальні в нормальному режимі. `awaiting_review` і `in_queue` різні семантично і дають різні вкладки в UI (PRD 8.3).

**`Act.status`** — 4 значення; конкретний набір переходів залежить від `Act.edo_provider` (snapshot, D-036):

```
edo_provider = dubidoc:
  draft → sent_to_edo → signed
                      ↘ deleted   (зворотний sync з Дубідок)

edo_provider = vchasno_external:
  draft → signed   (адмін явно натискає "Позначити підписаним" після підпису у Вчасно)
```

Для `vchasno_external` переходи `sent_to_edo` і `deleted` не застосовні (API-виклику в Дубідок немає, видалення ззовні нам невідоме). PDF можна перегенерувати в будь-якому статусі — канонічна підписана версія живе у Вчасно (D-036).

**`Payment.classification_reason`** — деталізує `awaiting_review`/`in_queue`:

| Reason                  | Належить до status | Опис                                                                                                                                                   |
| ----------------------- | ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `auto_act_disabled`     | `awaiting_review`  | Клієнт у списку виключень (D-003)                                                                                                                      |
| `external_edo`          | `awaiting_review`  | Клієнт має `edo_provider = vchasno_external`; авто-відправка через Дубідок не виконується, акт оформлюється вручну з підписом у Вчасно (D-036)         |
| `no_match`              | `in_queue`         | Клієнт не зматчений ні за номером договору, ні за ЄДРПОУ                                                                                               |
| `ambiguous_client`      | `in_queue`         | Номер договору вказує на одного клієнта, ЄДРПОУ платника — на іншого (D-009)                                                                           |
| `amount_mismatch`       | `in_queue`         | Сума не ділиться націло на ціну послуги                                                                                                                |
| `sms_quantity_mismatch` | `in_queue`         | Для СМС: parsed quantity не сходиться з amount                                                                                                         |
| `unknown_service_type`  | `in_queue`         | Не вдалось визначити тип послуги                                                                                                                       |
| `client_incomplete`     | `in_queue`         | Клієнт зматчений, але не заповнено обов'язкові поля (email, address, bank_name, bank_account, contract, apartments_count умовно) — D-017, D-019, D-022 |

Детальний список missing-полів для `client_incomplete` обчислюється UI на льоту з поточного стану клієнта, не зберігається в Payment (D-017).

### 3.4 Snapshot-поля в Act

В Act копіюються 5 категорій даних на момент генерації акту:

| Поле                         | Що копіює                                                                             | Підстава |
| ---------------------------- | ------------------------------------------------------------------------------------- | -------- |
| `unit_price`                 | Резолвлена ціна (override або тариф на дату платежу)                                  | D-010    |
| `quantity` + `quantity_unit` | Кількість одиниць послуги                                                             | D-010    |
| `client_snapshot` (jsonb)    | Реквізити Client: `name`, `legal_id`, `address`, `bank_name`, `bank_account`, `email` | D-004    |
| `contract_snapshot` (jsonb)  | `number` і `signed_date` договору                                                     | D-019    |
| `edo_provider`               | Канал ЕДО (`dubidoc` / `vchasno_external`), скопійований з `client.edo_provider`      | D-036    |

**Обґрунтування snapshot замість FK:**

- Акт — юридичний документ, прив'язаний до **факту оплати на конкретну дату**. Реквізити могли змінитись через рік; історичний акт має лишитись з тими реквізитами, що були вказані при підписанні.
- Зміна тарифу, реквізитів клієнта, номера договору або їхнє виправлення опечаток не повинні створювати ефекту "переписаної історії".
- При зміні ціни в період відкладеного платежу (9.10) snapshot гарантує, що акт фіксує ціну на дату платежу, не на дату пізнішої редакції.
- `edo_provider` snapshot гарантує, що акт, оформлений як Вчасно-документ, лишається таким навіть якщо клієнт згодом мігрує на Дубідок (і навпаки).

**Що НЕ snapshot:** посилання `payment_id` і `client_id` — це робочі FK для аналітики (фільтр актів по клієнту, навігація в UI). При зміні `Client.name` UI-список покаже нову назву, але PDF акту — стару.

### 3.5 Локальна БД і sync з "Моє ОСББ"

**Двигун:** PostgreSQL (D-024). Окремий інстанс, не та сама БД, що "Моє ОСББ". Підстава розділення — D-004 (різні життєві цикли контрагент-бухгалтерський vs клієнт-SaaS; ізоляція від основного продукту; snapshot-патерн).

**Зв'язок з "Моє ОСББ":** read-only sync з MySQL за розкладом `Settings.moeosbb_sync_schedule` (enum, D-023):

| Значення          | Поведінка                      | Примітка                                                                        |
| ----------------- | ------------------------------ | ------------------------------------------------------------------------------- |
| `first` (default) | Sync 1-го числа кожного місяця | Нові клієнти "Моє ОСББ" встигають потрапити в нашу БД до першого платежу місяця |
| `last`            | Sync останнього дня місяця     |                                                                                 |
| `manual`          | Автоматичного sync немає       | Тільки за кнопкою                                                               |

Незалежно від розкладу — у UI 8.7 є кнопка "Синхронізувати зараз" для разової операції (D-023).

**Що sync оновлює:** базові реквізити Client — `name`, `legal_id`, `address`, `bank_name`, `bank_account`, `email`.

**Що sync НЕ торкається** (manual-only — D-012, D-003, D-036):

- `apartments_count` — заповнюється вручну, бо в "Моє ОСББ" дані можуть бути неточними.
- `access_price_override` — наша система не має такого поняття в "Моє ОСББ".
- `auto_act_disabled` — наша система не має такого поняття в "Моє ОСББ".
- `edo_provider` — вибір каналу ЕДО (Дубідок / Вчасно-external) — наша система, не існує в "Моє ОСББ" (D-036).

**Критерій sync:** працює тільки для клієнтів, у яких `Client.moeosbb_user_id IS NOT NULL`. Решта (повністю локальні) ігноруються.

**Локальні клієнти без `moeosbb_user_id` (D-011):** адмін може створити клієнта в нашій БД безпосередньо — наприклад, якщо платіж прийшов від невідомого контрагента, який ще не заведений у "Моє ОСББ", або від клієнта поза SaaS. Такий клієнт є повноцінним: договір, акти, snapshot — все працює.

**Прив'язка локального клієнта до "Моє ОСББ" пізніше:** адмін вручну вписує `moeosbb_user_id` через картку клієнта в адмінці. Наступний sync почне оновлювати реквізити. Автоматичної прив'язки за ЄДРПОУ немає — рішення про відповідність приймає людина (D-011).

**Read-only гарантія:** ніяких write-операцій у БД "Моє ОСББ" система не виконує (D-001). Connection string у env-змінній `MOEOSBB_DB_URL` (D-020); рекомендується видавати read-only MySQL-користувача (10.3).

## 4. Класифікація платежів

Класифікація — це детермінований алгоритм, що для кожного нового `Payment` визначає `client_id`, `service_type`, `unit_price`, `quantity` і вирішує, чи можна одразу згенерувати акт. Точний псевдокод — у `domain-model.md § Логіка класифікації платежу`. Тут — продуктовий огляд, обґрунтування кожного кроку і відсилки до edge cases.

### 4.1 Загальна логіка

Класифікація запускається на кожному `Payment` зі статусом `received` — або одразу після polling-у (4.7), або повторно після ручної корекції в UI 8.3. Кроки:

| #   | Крок                                                                                                          | Можливі виходи                                                                                                                                                                                |
| --- | ------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Парсинг номера договору** з `payment.purpose` через regex-патерни                                           | список номерів (можливо порожній)                                                                                                                                                             |
| 1b  | **Dedup + перевірка кількості** (D-027)                                                                       | `in_queue(multiple_contracts)` якщо >1 різних номерів                                                                                                                                         |
| 2   | **Матчинг клієнта** через номер договору і ЄДРПОУ платника (D-009) з винятком для транзитного рахунку (D-008) | `client` визначено, або `in_queue(no_match \| ambiguous_client)`                                                                                                                              |
| 3   | **Перевірка `auto_act_disabled`** (D-003)                                                                     | `awaiting_review(auto_act_disabled)` якщо true                                                                                                                                                |
| 3b  | **Перевірка `edo_provider`** (D-036)                                                                          | `awaiting_review(external_edo)` якщо `client.edo_provider = vchasno_external`. При одночасній установці з `auto_act_disabled` пріоритет має `auto_act_disabled` (вказує на проблему з даними) |
| 4   | **Визначення типу послуги** за ключовими словами в purpose                                                    | `access` або `sms`. `other` — тільки вручну (D-016)                                                                                                                                           |
| 5   | **Перевірка повноти даних клієнта** (D-017, D-019, D-022) — до резолву ціни                                   | `in_queue(client_incomplete)` з missing-списком, обчислюваним UI на льоту                                                                                                                     |
| 6   | **Резолв ціни** (override → Tariff / SmsPrice) — див. 5.4                                                     | `unit_price` визначено                                                                                                                                                                        |
| 7   | **Визначення quantity**                                                                                       | `amount_mismatch` / `sms_quantity_mismatch` або `quantity` визначено                                                                                                                          |
| 8   | **Генерація акту** і фіксація snapshot-ів                                                                     | `Payment.status=classified`, `Act` створено                                                                                                                                                   |

Логіку перебудовано в порядку, що мінімізує "роботу даремно": спочатку дешеві перевірки (regex, lookup), потім ідентифікація і повнота даних, потім ціна, потім фінальна валідація amount.

### 4.2 Парсинг номера договору

**Стартовий набір regex-патернів** (зберігається в `Settings.contract_regex_patterns`, редагується з UI 8.7 без перевипуску коду):

```
1. /договір\s*[№#N]?\s*(\d{5,6})/i                # "договір №556770"
2. /дог[оi]в[оi][рp]\s*[№#N]?\s*(\d{5,6})/i       # опечатки "договiр", "договiо"
3. /dogovir\s*[№#N]?\s*(\d{5,6})/i                # латиниця "dogovir 556770"
4. /(\d{6})\s+ЗГІДНО\s+ДОГОВОРУ/i                 # "556434 ЗГІДНО ДОГОВОРУ"
5. /[№#N]\s*(\d{5,6})(?!\d)/i                     # просто "№556770" без слова
```

Покриття патернів — близько 80% спостережених варіантів з `research/payments-analysis.md`. Нестандартні формати (наприклад, `WB-4`, `011220`) додаються окремими патернами в Settings у міру виявлення.

**Кілька знайдених номерів (D-027):** після прогону всіх патернів результати дедуплікуються (унікальні значення). Якщо після dedup залишилось:

- `0` номерів → переходимо до матчингу по ЄДРПОУ;
- `1` номер → нормальний матчинг;
- `>1` різних номерів → `in_queue(multiple_contracts)`. Адмін у UI 8.3 бачить список знайдених номерів і обирає правильний (або позначає платіж як `skipped`, якщо платіж реально стосується кількох клієнтів — таке розщеплення поза скоупом, D-007).

**Обробка опечаток і латиниці:** через окремі patterns (2, 3 вище). Це свідомий вибір замість fuzzy-матчингу — кожен pattern легко інспектувати, кожна нова форма опечатки додається явним правилом.

### 4.3 Матчинг клієнта

Базова стратегія — **двофакторний матчинг** (D-009): номер договору + ЄДРПОУ платника. Це закриває 8 виявлених в виписці випадків, коли один номер договору з'являється з різних ЄДРПОУ (помилка платника, дубль, помилка розпізнавання).

**Алгоритм:**

1. Якщо `payer_legal_id ∈ Settings.transit_edrpou_list` (D-008) — це транзитний рахунок банку. ЄДРПОУ платника — це ЄДРПОУ банку, реальний платник у тексті після коми. Двофакторну перевірку **пропускаємо**: матчимо тільки за номером договору. Якщо номера немає або не знайдено клієнта — `in_queue(no_match)`. Fuzzy-матч по ПІБ — не в MVP.
2. Нормальний платіж з номером договору:
   - lookup → знайдено клієнта → перевірка `client.legal_id == payer_legal_id`:
     - **збіг** → клієнт визначено;
     - **розбіжність** → `in_queue(ambiguous_client)`.
   - lookup → не знайдено → fallback на матчинг по ЄДРПОУ (можливо, клієнт є, але номер у purpose з опечаткою або в незнайомому форматі):
     - знайдено → клієнт визначено;
     - не знайдено → `in_queue(no_match)`.
3. Нормальний платіж без номера договору — тільки lookup по ЄДРПОУ. Якщо не знайдено → `in_queue(no_match)`.

**Fuzzy-матч по назві клієнта** (як підказка в адмінці) — поза MVP. У майбутньому можна показувати на картці платежу в `in_queue` "схожі клієнти", але це покращення UX, не зміна логіки.

### 4.4 Визначення типу послуги

Автокласифікація повертає **тільки** `access` або `sms` (D-016). Логіка:

- Якщо в `purpose.lower()` зустрічається хоча б одне ключове слово зі `Settings.sms_keywords` (стартово: `смс`, `sms`, `повідомлення`) → `sms`.
- Інакше → `access` (default).

`service_type=other` присвоюється виключно вручну в UI (D-016), для нестандартних послуг (консультації, налаштування тощо). Авто-класифікатор не може вибрати `other` за дизайном — це уникає false-positive class assignments.

`unknown_service_type` як reason — резерв на випадок, якщо в майбутньому з'являться кейси, де ні `access`, ні `sms` не підходять, і навіть default `access` не правильний. У стартовій логіці цей reason не повертається.

### 4.5 Визначення quantity

**Для `access`:**

```
IF payment.amount % unit_price != 0:
    return in_queue(reason=amount_mismatch)
quantity = payment.amount / unit_price
quantity_unit = "міс."
```

Це покриває нормальний кейс (200 грн → 1 міс), оплату наперед (1000 грн → 5 міс) і доплату (400 грн → 2 міс). Дробні значення (наприклад, 350 грн при ціні 200) ловляться `amount_mismatch`.

**Для `sms`:**

```
parsed_quantity = parse_sms_quantity_from_purpose(payment.purpose)
IF parsed_quantity * unit_price != payment.amount:
    return in_queue(reason=sms_quantity_mismatch)
quantity = parsed_quantity
quantity_unit = "шт."
```

Quantity для СМС не виводиться з `amount / unit_price` (це дало б дробні значення при копійках округлення). Парсинг з тексту призначення (`"у кількості 100"`) дає явну цифру; верифікація `parsed_quantity × unit_price == amount` — захист від помилок парсингу або платіжки. Деталі patterns для `parse_sms_quantity_from_purpose` уточнюються при першій реалізації, орієнтуючись на `research/payments-analysis.md`.

### 4.7 Idempotency

Дві точки, де треба гарантувати "один платіж = один акт, навіть при повторному запуску":

**A. Polling ПриватБанку → запис у `Payment`.**

- `Payment.bank_transaction_id` має UNIQUE constraint.
- INSERT виконується як `INSERT ... ON CONFLICT (bank_transaction_id) DO NOTHING`.
- Polling може брати overlapping період (наприклад, "за останні 2 години" при інтервалі 1 година) — DB гарантує, що дублювання не відбудеться.
- `raw_data` при повторному отриманні **не оновлюється**: вважаємо першу версію канонічною. Якщо ПриватБанк виправив транзакцію — це ручний кейс для адміна (рідкісний).

**B. Класифікація `Payment` → створення `Act`.**

- Класифікатор працює тільки на `Payment.status = received` (або після ручного reclassify — `awaiting_review` / `in_queue`).
- Вся класифікація обгорнута в транзакцію PostgreSQL: створення `Act` + `UPDATE Payment SET status=classified, act_id=...` атомарні.
- Якщо транзакція впала між створенням Act і встановленням `Payment.act_id` — обидва відкочуються, платіж лишається `received`, наступний запуск спрацює нормально.
- Захист від подвійної класифікації: при старті класифікатор робить `SELECT ... FOR UPDATE` на `Payment` — паралельні запуски однієї транзакції чекають.

**Відправка в EDO (Дубідок)** — окремий етап з власною idempotency (PRD 7.2 і 9.7). Не блокує idempotency класифікації.

## 5. Цінова політика і тарифікація

Цінова політика — двошарова: **тарифна сітка** (для послуги "доступ") і **глобальна історія ціни СМС** як окрема сутність. На рівні клієнта є **override** для виняткових випадків. У момент генерації акту ціна резолвиться і фіксується snapshot-ом (3.4).

### 5.1 Тарифна сітка (Tariff) — для послуги "доступ"

Тарифна сітка моделюється сутністю `Tariff` з полями `apartments_min`, `apartments_max` (NULL = catch-all "і більше"), `price`, `effective_from`. CRUD доступний з UI 8.7 з самого старту системи (D-013) — навіть якщо на момент запуску в сітці лише одне правило.

**Інваріант (D-018):** сітка завжди містить рівно **одне catch-all правило** з `apartments_max IS NULL`. Якщо адмін намагається видалити catch-all або залишити сітку без жодного — CRUD блокує операцію.

**Стартова конфігурація:** одне правило `{apartments_min=0, apartments_max=NULL, price=200, effective_from=<дата запуску>}` — еквівалент "200 грн для всіх". Власник може диференціювати ціни за розміром ОСББ, додавши ranged-правила.

**Версіонування у часі (`effective_from`):** при зміні ціни не редагуємо існуючий рядок — додаємо новий з пізнішим `effective_from`. Старе правило залишається для коректного резолву історичних платежів.

**Пріоритет при множинному матчингу** (D-018): ranged правила перед catch-all → серед однакового типу вужчий діапазон перед ширшим → серед однакового діапазону свіжіше `effective_from` перед старішим.

### 5.2 Глобальна історія ціни СМС (SmsPrice) — D-026

Симетрично до Tariff, але без критерію розміру: ціна СМС однакова для всіх клієнтів і змінюється тільки в часі. Сутність `SmsPrice` з полями `price`, `effective_from`. CRUD з UI 8.7.

**Стартова конфігурація:** один рядок `{effective_from='2024-01-01', price=1.40}`.

**Чому окрема таблиця, а не Settings:** Settings — це key-value поточних значень. Для коректного резолву ціни на дату платежу (особливо в сценарії 9.10 — відкладений платіж після зміни ціни) потрібна історизація з `effective_from`, що структурно не вкладається в key-value. Симетрія з Tariff робить рішення прозорим.

**Резолв:** `resolve_sms_price(payment_date)` повертає `price` найсвіжішого правила з `effective_from <= payment_date`.

`Settings.sms_min_quantity` (мінімальне замовлення СМС, 100 шт) лишається в Settings — це бізнес-правило, не цінова історія.

### 5.3 Override на клієнта

`Client.access_price_override` (`numeric(10,2)`, nullable) — виняток із сітки для конкретного клієнта. Якщо задано — застосовується тільки до `service_type=access`, перебиваючи сітку повністю (не "знижка від базової", а "інша ціна замість").

**Сценарії використання:**

- Великий ОСББ з індивідуальною домовленістю про знижену ціну.
- ФОП-управитель з кількома ОСББ і окремим тарифом.
- Тимчасові акції, які не вмонтовуються в сітку через ranged-структуру.

**Що override НЕ робить:**

- Не впливає на СМС-ціну (для цього довелось би вводити `Client.sms_price_override` — поки не потрібно).
- Не керує авто-генерацією акту — за це відповідає `auto_act_disabled` (ортогональна семантика).
- Не версіонується (зберігається тільки поточне значення; при зміні старі акти захищені snapshot-ом).

### 5.4 Резолв ціни (algorithm)

Псевдокод для платежу з відомим `client`, `service_type` і `payment.date`:

```
resolve_unit_price(client, service_type, payment_date):
    IF service_type == sms:
        return resolve_sms_price(payment_date)

    IF service_type == access:
        # 1. Override має пріоритет
        IF client.access_price_override IS NOT NULL:
            return client.access_price_override

        # 2. Тарифна сітка
        # apartments_count гарантовано NOT NULL для access без override (D-017,
        # перевірено в кроці 5 класифікації).
        # Catch-all правило гарантовано існує (D-018), тому tariff_lookup
        # завжди повертає рядок.
        return tariff_lookup(client.apartments_count, payment_date).price

    IF service_type == other:
        # Ціна задається вручну в акті (D-016); ця функція не викликається.
        raise NotApplicable
```

Точна реалізація `tariff_lookup` — у `domain-model.md § Логіка резолву ціни` (з SQL ORDER BY для пріоритету).

### 5.5 Snapshot ціни в акт

При генерації акту резолвлена ціна копіюється в `Act.unit_price` як snapshot (D-010). Після цього зміни в Tariff, SmsPrice або `access_price_override` НЕ переписують `Act.unit_price` — історичні акти лишаються незмінними.

**Чому це критично:**

- Юридично акт фіксує суму на дату платежу. Якщо ціна змінилась через рік, перерахунок — окрема операція з окремим документом, не правка старого акту.
- Помилка в тарифі (наприклад, неправильно введене число) виправляється новим правилом у сітці з `effective_from = вчора`, не редагуванням старого. Старі акти лишаються такими, якими були підписані.
- Snapshot ціни консистентний з другими snapshot-полями (`client_snapshot`, `contract_snapshot`, `quantity`) — див. 3.4.

### 5.6 Глобальні налаштування цін (Settings)

Що лишилось у Settings після D-026 і D-020:

| Ключ                       | Призначення                                                                                              |
| -------------------------- | -------------------------------------------------------------------------------------------------------- |
| `sms_min_quantity`         | Мінімальне замовлення СМС (100 шт). Використовується для валідації в адмінці; до резолву ціни не входить |
| `polling_interval_minutes` | Інтервал polling ПриватБанку (60 хв за замовч.)                                                          |
| `transit_edrpou_list`      | Список ЄДРПОУ транзитних рахунків (D-008) — впливає на класифікацію, не на ціну                          |
| `contract_regex_patterns`  | Regex-патерни для парсингу номера договору                                                               |
| `sms_keywords`             | Ключові слова для класифікації СМС                                                                       |
| `moeosbb_sync_schedule`    | Розклад sync з "Моє ОСББ" (D-023)                                                                        |

**Чого тут НЕМАЄ:** жодних цін і `effective_from` — все цінове перенесено в `Tariff` і `SmsPrice`. Жодних credentials — все в env (D-020).

## 6. Генерація актів

Генерація акту — це фінальний крок успішної класифікації платежу (4.1, крок 8) або ручної дії адміна над `awaiting_review`/`in_queue` платежем. Включає: визначення номера, рендер PDF, фіксацію snapshot-полів, відправку в EDO. Реквізити Дубідок як EDO винесені в розділ 7.2; тут — лише локальні правила.

### 6.1 Правила генерації

| Правило                                                                                                                                                                                                                                            | Підстава            |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------- |
| **Один платіж = один акт** (cardinality 1:1)                                                                                                                                                                                                       | D-007               |
| **Дата акту = останній календарний день місяця платежу**                                                                                                                                                                                           | D-006               |
| **Тригер автогенерації:** платіж успішно класифікований (`Payment.status` переходить у `classified`) для клієнта з `auto_act_disabled = false` І `edo_provider = dubidoc`                                                                          | D-003, D-014, D-036 |
| **Договір обов'язковий перед генерацією акту** (для всіх `service_type`)                                                                                                                                                                           | D-019               |
| **Service_type = `other` присвоюється тільки вручну** і вимагає ручного заповнення `unit_price`, `quantity`, `service_description`                                                                                                                 | D-016               |
| **`Act.edo_provider` як snapshot:** копіюється з `client.edo_provider` при створенні акту. Для `dubidoc` запускається відправка в Дубідок API; для `vchasno_external` API не викликається, акт лишається `draft` до ручного "Позначити підписаним" | D-036               |

**Що блокує авто-генерацію** (відбивається в крок 3/3b/5 класифікації — 4.1):

- `client.auto_act_disabled = true` → `awaiting_review(auto_act_disabled)`. Адмін генерує акт вручну з підтвердженням параметрів.
- `client.edo_provider = vchasno_external` → `awaiting_review(external_edo)`. Адмін генерує акт вручну, акт відразу створюється з `Act.edo_provider = vchasno_external` і без виклику Дубідок API (D-036).
- `client.contract IS NULL` → `in_queue(client_incomplete, missing=[contract])`. Адмін заводить договір, потім реруниться класифікація.
- Будь-який інший `client_incomplete`, `amount_mismatch`, `ambiguous_client` тощо → `in_queue`. Аналогічно.

**Дата акту незалежно від дати створення в БД.** Платіж від 5 квітня обробляється відразу після polling-у (наприклад, 5 квітня 10:00), але `act_date = 2026-04-30`. Підпис у Дубідок здійснюється не раніше `act_date` (юридична вимога — D-006).

### 6.2 Нумерація

**Формат (D-005):**

- Перший акт для клієнта в місяці M: `№M` (наприклад, `№5` за травень).
- Другий і далі: `№M/N`, де N — порядковий номер у місяці (`№5/2`, `№5/3`).
- Нумерація **окрема для кожного клієнта**.

**Алгоритм:**

```
generate_next_act_number(client_id, act_year, act_month):
    existing_count = SELECT count(*) FROM acts
                     WHERE client_id = $client_id
                       AND extract(year FROM act_date) = $act_year
                       AND extract(month FROM act_date) = $act_month
                     FOR UPDATE   -- (1) lock для race-safe інкременту
    IF existing_count == 0:
        return f"№{act_month}"
    ELSE:
        return f"№{act_month}/{existing_count + 1}"
```

**Конкурентність:**

- Генерація акту відбувається в межах транзакції класифікації (4.7).
- `FOR UPDATE` на acts по `(client_id, year, month)` блокує паралельні класифікації для того самого клієнта в тому самому місяці.
- Унікальний індекс `(client_id, act_date, number)` — додатковий захист на рівні БД: якщо логіка інкременту дала збій і два процеси спробують записати `№5/2` → один отримає `UNIQUE violation`, транзакція відкотиться, retry на новому інкременті.
- На практиці race-conditions малоймовірні (single admin, рідкісні платежі), але захист є на двох рівнях.

**Чому не глобальна нумерація:** D-005. Окрема серія на клієнта — поточна практика автора, відповідає очікуванням клієнтів-ОСББ і їхньої бухгалтерії, не потребує глобального counter-а.

### 6.3 Шаблони і рендер PDF (D-028)

**Архітектура рендеру:**

- PDF генерується **локально** в нашій системі. У Дубідок передається готовий файл як base64 (`POST /api/v1/documents`, поле `file`).
- Хмарне API htmldocs.com **не використовується** — реквізити клієнтів (ЄДРПОУ, адреса, IBAN, email, сума) не виходять за периметр сервера (D-028).
- Шаблон акту — **HTML + React + Tailwind** (htmldocs DX або еквівалент): один stack з web-адмінкою (8.x), WYSIWYG-preview, легка правка юридичних формулювань без перебудови коду PDF-генератора.
- Конкретний рантайм рендеру (Puppeteer / Playwright / `@htmldocs/render` self-hosted / альтернатива) — деталь 10.1; принципова вимога — все у власній інфраструктурі.

**Шаблони:**

| `service_type` | Опис                         | `service_description` (рендериться у "Найменування послуги")                                                                          |
| -------------- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| `access`       | Доступ до сервісу "Моє ОСББ" | Автогенерація з шаблону: `"Доступ до сервісу за період {quantity} {quantity_unit}"` (наприклад, "Доступ до сервісу за період 1 міс.") |
| `sms`          | СМС-розсилка                 | Автогенерація: `"СМС-розсилка {quantity} {quantity_unit}"` (наприклад, "СМС-розсилка 100 шт.")                                        |
| `other`        | Нестандартна послуга         | **Заповнюється вручну** при створенні акту. `service_description` обов'язкове (D-016)                                                 |

Адмін може **відредагувати `service_description`** у картці акту перед відправкою в EDO (статус `draft`). Після `sent_to_edo` правка не блокується на рівні БД, але створює ризик розсинхронізації з тим, що в Дубідок — рекомендується робити це лише до відправки. Для актів з `edo_provider = vchasno_external` (D-036) правка і перегенерація PDF дозволені в будь-якому статусі — канонічна версія живе у Вчасно, локальний PDF є допоміжною копією.

**Поля, що рендеряться в PDF:**

- Шапка: ФОП-виконавець (з env/config, не з БД), реквізити ФОПа.
- Реквізити Замовника: з `Act.client_snapshot` (name, legal_id, address, bank_name + bank_account як один рядок "Поточний рахунок: {bank_account} в {bank_name}").
- Договір: з `Act.contract_snapshot` ("по договору №{number} від {signed_date}").
- Таблиця послуги: `service_description`, `quantity` + `quantity_unit`, `unit_price`, `total_amount`.
- Підвал: підписи сторін (порожні поля для підпису), дата = `act_date`.

**Зразок:** `samples/acts/act-2026-04_200.pdf` (доступ), `samples/acts/2026-02 Akt SMS.pdf` (СМС). Юридичні формулювання і структура мають відповідати цим зразкам.

### 6.4 Snapshot реквізитів і ціни

При генерації акту копіюються 5 категорій даних (повторюємо з 3.4 для повноти розділу):

| Snapshot                         | Що копіює                                                                      | Підстава |
| -------------------------------- | ------------------------------------------------------------------------------ | -------- |
| `Act.unit_price`                 | Резолвлена ціна (override → Tariff / SmsPrice — див. 5.4)                      | D-010    |
| `Act.quantity` + `quantity_unit` | Кількість одиниць послуги                                                      | D-010    |
| `Act.client_snapshot`            | `name`, `legal_id`, `address`, `bank_name`, `bank_account`, `email` з `Client` | D-004    |
| `Act.contract_snapshot`          | `number` + `signed_date` з `Contract`                                          | D-019    |
| `Act.edo_provider`               | Канал ЕДО (`dubidoc` / `vchasno_external`) з `Client.edo_provider`             | D-036    |

**Поведінка при пізнішій зміні реквізитів клієнта:**

- Зміна `Client.name` / `legal_id` / `address` / `bank_name` / `bank_account` / `email` (через sync з "Моє ОСББ" або ручне редагування) **НЕ перезаписує** історичні `Act.client_snapshot`.
- Перевипуск акту з новими реквізитами — окрема ручна дія в UI (8.6), яка створює новий `Act` як скоригований документ. Старий лишається в БД зі статусом `deleted` (після видалення в Дубідок — див. 9.5, 9.9).
- Аналогічно для `Contract.number` / `signed_date` — `Act.contract_snapshot` immutable.
- Tariff / SmsPrice — при зміні `unit_price` в `Act` фіксує ціну на дату резолву (= дату класифікації платежу). Зміна ціни в системі заднім числом не торкає історичні акти. Детальніше — 9.10.

**Чому snapshot замість FK на реквізити:** юридична незмінність акту. Дивись повне обґрунтування в 3.4 і D-004, D-010, D-019.

## 7. Інтеграції

Три зовнішніх системи з API: **ПриватБанк Автоклієнт API** (вхідні платежі), **Дубідок API** (відправка актів, statuses), **MySQL "Моє ОСББ"** (read-only sync реквізитів клієнтів). Окремо — **Вчасно** як зовнішнє ЕДО без API-інтеграції (D-036): ручний експорт PDF з нашої адмінки і ручний підпис у Вчасно поза системою. Дві API-інтеграції описано у відповідних D-записах (D-020, D-021, D-023, D-029); тут — продуктовий огляд і операційні правила.

### 7.1 ПриватБанк (Автоклієнт API)

**Що використовуємо:**

- Endpoint `GET` транзакцій за період (точна назва — у Privatbank_API.pdf, передається при імплементації 10.1).
- Реюз існуючого коду з `~/Projects/privatbank-telegram-bot` (модуль HTTP-клієнта, парсинг payload).

**Polling-стратегія:**

- Інтервал — `Settings.privatbank_polling_interval_minutes` (default 60).
- Кожен запит бере **overlapping період** (наприклад, останні 2 години при інтервалі 1 година) — щоб переживати разові пропуски через мережеві збої.
- Дедуплікація гарантована UNIQUE-constraint на `Payment.bank_transaction_id` (4.7): `INSERT ... ON CONFLICT (bank_transaction_id) DO NOTHING`.

**Поля, що витягуємо з API і мапимо на `Payment`:**
| Поле API | Поле моделі |
|---|---|
| ID транзакції | `bank_transaction_id` |
| Дата | `payment_date` |
| Сума | `amount` |
| Призначення | `purpose` |
| Назва платника | `payer_name` |
| ЄДРПОУ/РНОКПП платника | `payer_legal_id` |
| IBAN платника | `payer_bank_account` |
| Повний payload | `raw_data` (jsonb) |

**Обробка помилок:**

- `401 Unauthorized` → токен невалідний/протух. Polling зупиняється до перевипуску token (env `PRIVATBANK_TOKEN`). Банер у адмінці (10.4, D-030).
- `5xx` / network errors → 3 retries з exponential backoff (1с / 5с / 30с). Якщо не вдалось — пропускаємо цикл, наступний polling зчитає overlapping період і досягне eventual consistency.
- `429 Too Many Requests` → respect Retry-After header.
- Будь-яка persistent помилка (4 послідовних провали) → банер у адмінці з типом помилки і часом останнього успішного polling-у.

**Idempotency:** див. 4.7. UNIQUE по `bank_transaction_id` + `ON CONFLICT DO NOTHING`. `raw_data` при повторі НЕ оновлюється (перша версія канонічна).

**Secrets:** токен у env `PRIVATBANK_TOKEN` (D-020). UI 8.7 не редагує токен, тільки показує read-only статус підключення.

### 7.2 Дубідок (D-029)

**Застосовується тільки для актів з `Act.edo_provider = dubidoc`** (D-036). Для актів з `edo_provider = vchasno_external` ніяких API-викликів у Дубідок не робиться — див. 7.4.

**Тариф:** обов'язковий **Premium-план** (підтверджено власником). Без Premium недоступні поля `date`, `number`, `amount`, `participants[]`, що ламає авто-відправку акту клієнту після підпису власником.

**Операції, що використовуємо:**
| Endpoint | Коли |
|---|---|
| `POST /api/v1/documents` | При генерації акту (`Act.status` → `sent_to_edo`) |
| `GET /api/v1/documents/{id}` | Polling статусу акту (раз в `dubidoc_poll_interval_hours`, default 6) |
| `GET /api/v1/documents/{id}/download` | За потреби — для збереження підписаного PDF у наш storage |

Інші endpoint'и Дубідок (`/api/v1/contacts`, `/api/v1/documents/{id}/send`, `/api/v1/folders`, `/api/v1/tags`, archive endpoints) — **НЕ використовуються в MVP**.

**Передача реквізитів контрагента — inline:**

- Окремий sync довідника `/api/v1/contacts` **не виконується**. Дубідок сам автоматично створює запис у своєму довіднику з `participants[]` при першому використанні.
- При `POST /documents` передаємо в `participants[]` один елемент:
  ```json
  {
    "action": "sign",
    "email": "<client_snapshot.email>",
    "edrpou": "<client_snapshot.legal_id>",
    "priority": 1,
    "isSignatureRequired": true
  }
  ```
- Підпис власника (ФОПа) робиться окремою ручною операцією через UI Дубідок (D-001 — ручний підпис).
- Поля документа: `file` (base64 PDF), `title` ("Акт виконаних робіт №M клієнт Y"), `filename` ("act-{id}-{number}.pdf"), `date` = `Act.act_date`, `number` = `Act.number`, `amount` = `Act.total_amount` (округлено до int — Дубідок чекає integer для amount).
- `signatureType = "external"` (default) — підписувач поза організацією.
- `workflowType = "sequential"` — спочатку власник підписує, потім клієнт.
- `callbackUrl` НЕ передається (polling-only — див. нижче).

**Polling статусів (D-029):**

- Cron-задача раз на `dubidoc_poll_interval_hours` (default 6) бере всі `Act` зі статусом `sent_to_edo` **і `edo_provider = dubidoc`** (D-036) і робить `GET /api/v1/documents/{id}` для кожного.
- Мапінг відповіді → наша модель:
  | Дубідок | Наша модель |
  |---|---|
  | `status = "signed"` | `Act.status = signed` |
  | `archived = true` | `Act.status = deleted` (Дубідок не має explicit delete; archive — еквівалент). Платіж `Payment.act_id` встановлюється у `NULL` (D-025), стає доступним для повторної класифікації — 9.9 |
  | `refused = true` | `Act.status` лишається `sent_to_edo`, але `Act.edo_status` фіксує `"refused"`. UI 8.6 показує індикатор "клієнт відмовився" |
  | інші Дубідок-статуси | `Act.edo_status = "<raw>"` (text), `Act.status = sent_to_edo` |
- `Act.edo_status` — `text`, не ENUM (D-029): Дубідок-набір з 15+ значень занадто широкий і нестабільний для нашої типізації.

**Чому polling, не webhook (D-029):**

- Polling прибирає вимогу публічного HTTPS endpoint для callback — простіший деплой і безпека.
- Затримка ≤6 годин в виявленні підпису прийнятна для single-admin сценарію: підпис не тригерить жодних автоматичних дій нашої системи (клієнту акт надсилається автоматично Дубідок-ом, незалежно від нас).
- Webhook як майбутнє покращення — Phase 2 (PRD 11), якщо адмін захоче бачити підписи негайно.

**Обробка помилок:**

- `401` → токен невалідний. Зупинка polling + відправок до перевипуску `DUBIDOC_TOKEN` (env). Банер.
- `5xx` / network → 3 retries з backoff.
- `429` → respect Retry-After.
- При відправці акту (`POST /documents`) — якщо постійна помилка, `Act.status` лишається `draft`, акт в UI 8.6 з індикатором "не відправлено", адмін може ретригерити вручну. Див. 9.7.

**Реюз коду:** `~/Projects/zbory_v2` має готового HTTP-клієнта і обробку webhook (зараз не використовуємо webhook). Беремо HTTP-клієнт і парсинг.

**Secrets:** токен у env `DUBIDOC_TOKEN`. UI 8.7 показує тільки read-only статус.

### 7.3 "Моє ОСББ" (MySQL, read-only)

**Що читаємо:**

- Таблиця `osbb_users` (D-021) — назва, ЄДРПОУ, адреса, реквізити банку, email.
- Точний перелік полів і їх мапінг на `Client` — `domain-model.md § Client` (синхронізуються `name`, `legal_id`, `address`, `bank_name`, `bank_account`, `email`).

**Періодичність (D-023):**

- Автоматично — за розкладом `Settings.moeosbb_sync_schedule`:
  - `first` (default) — 1-го числа місяця
  - `last` — останній день місяця
  - `manual` — без автоматичного sync
- Ручна кнопка "Синхронізувати зараз" у UI 8.7 — доступна завжди, незалежно від розкладу.

**Критерій sync:** тільки `Client.moeosbb_user_id IS NOT NULL`. Локальні клієнти (D-011) ігноруються.

**Що НЕ перезаписується sync-ом:**

- `apartments_count` — manual only (D-012).
- `access_price_override` — наша система, не існує в "Моє ОСББ".
- `auto_act_disabled` — наша система, не існує в "Моє ОСББ".

**Обробка зміни реквізитів** (наприклад, ОСББ змінило адресу або IBAN):

- Sync оновить `Client.address` / `Client.bank_account` / тощо.
- Існуючі акти лишаються незмінними (snapshot — 3.4, 6.4).
- Адмін бачить нові реквізити в картці клієнта (8.4). Якщо потрібно перевипустити старий акт з новими реквізитами — це окрема дія в UI 8.6 (створення скоригованого акту), не автоматична.
- Sync не повідомляє адміну про конкретні дельти — він просто оновлює. Якщо буде корисно бачити "що змінилось при останньому sync" — окреме рішення в Phase 2.

**Read-only гарантія (D-001):**

- Жодних write-операцій у БД "Моє ОСББ". Сync — тільки `SELECT`.
- Рекомендується видавати окремого read-only MySQL-користувача для `MOEOSBB_DB_URL` (10.3).
- Connection string у env `MOEOSBB_DB_URL` (D-020); UI 8.7 показує read-only статус підключення.

### 7.4 Вчасно (зовнішнє ЕДО, без API-інтеграції) — D-036

**Призначення:** канал оформлення актів для клієнтів, які категорично проти Дубідок або мають внутрішні вимоги до конкретного ЕДО (Вчасно — vchasno.ua, від EVO). Активується встановленням `Client.edo_provider = vchasno_external` у картці клієнта (8.4). Інший EDO-провайдер чи паперовий процес — поза скоупом MVP (одне зарезервоване значення enum).

**Workflow** (на відміну від Дубідок-каналу, повністю ручний):

1. Платіж від такого клієнта проходить класифікацію (4.1), всі перевірки даних (4.5, 4.3, 5.4 тощо). Крок 3b повертає `awaiting_review(external_edo)` (4.6).
2. У черзі розбору (8.3, вкладка "На апрув") адмін бачить платіж, перевіряє автокласифікацію, при потребі коригує параметри.
3. Адмін натискає "Згенерувати акт". Створюється `Act` з:
   - `edo_provider = vchasno_external` (snapshot з клієнта).
   - `status = draft`.
   - `edo_doc_id = NULL`, `edo_status = NULL` (поля polling-а Дубідок, не застосовні).
   - Згенерований PDF у `pdf_file_url`.
4. **API Дубідок НЕ викликається.** Акт лишається в `draft` до явної дії адміна.
5. Адмін викачує PDF з картки акту (8.6), завантажує у Вчасно (vchasno.ua), ставить підпис там. Цей крок повністю поза нашою системою.
6. Адмін повертається в адмінку, натискає "Позначити підписаним" → `Act.status = signed`. Перехід `draft → signed` (D-036, state machine).
7. PDF можна перегенерувати в будь-якому статусі — канонічна підписана версія живе у Вчасно (див. 6.3, 6.4).

**Чого тут НЕМАЄ:**

- Інтеграції з Вчасно API (поза скоупом D-036). Якщо в майбутньому з'явиться потреба автоматизувати — окреме рішення з підняттям повноцінного multi-provider в моделі.
- Авто-розсилки підписаного акту клієнту з нашого боку. У Вчасно є власні механізми доставки/архівації — це справа адміна і Вчасно.
- Polling статусів. Сторона Вчасно нічого нам не повідомляє; `Act.status = signed` встановлюється виключно ручною дією адміна (4.6).
- Поле `Settings.vchasno_*` чи токени Вчасно. Жодних credentials.

**Що в UI 8.6:**

- Бейдж "Вчасно" в переліку актів (фільтр по `edo_provider`).
- Кнопка "Позначити підписаним" замість "Перейти в Дубідок" / "Оновити статус з Дубідок".
- Кнопка "Перегенерувати PDF" доступна навіть для `status = signed`.

**Що в UI 8.4 (Клієнти):**

- Поле `edo_provider` як select (`Дубідок` / `Вчасно (зовнішнє)`), default `Дубідок` для нових клієнтів.
- При зміні значення warning: "Зміна каналу ЕДО НЕ змінює вже згенеровані акти — у них фіксується snapshot. Нові акти будуть оформлюватись за новим каналом" (D-036).

**Обмеження зі сторони бізнес-логіки:**

- Алгоритм класифікації (4.1, 4.2, 4.3, 5.4) і генерації акту (6.1, 6.2, 6.3) однаковий для обох каналів — змінюється тільки фінальний крок (відправити в Дубідок vs не відправляти).
- Нумерація актів (6.2) — спільна для обох каналів: `№M / №M/N` у межах клієнта в межах місяця, незалежно від `edo_provider`.

## 8. Адмін-інтерфейс

Адмінка — основний (єдиний — D-001) інтерфейс взаємодії автора з системою. Стек: Next.js (App Router) + Tailwind + shadcn/ui (D-031). Дизайн-принцип — давати адміну швидкий доступ до того, де **потрібне його втручання** (черга розбору, статус системи), і простий доступ до даних (списки, картки) для нечастих операцій.

### 8.1 Інформаційна архітектура

**Структура навігації** (sidebar):

```
Дашборд              /
Платежі              /payments
Черга розбору        /queue        ← badge з кількістю
Клієнти              /clients
Договори             /contracts
Акти                 /acts
Тарифи               /tariffs
Налаштування         /settings
```

**Дашборд (`/`):**

- Банери стану системи (D-030): ПриватБанк, Дубідок, "Моє ОСББ" — для кожного: ✓ ОК / ✗ Помилка + час останньої успішної взаємодії.
- Лічильники: "Платежів у черзі: X", "Платежів на апрув: Y", "Актів очікують підпису: Z".
- Останні 10 платежів (зведений вид зі статусом).
- Кнопки: "Синхронізувати ПриватБанк зараз", "Синхронізувати Моє ОСББ зараз", "Опитати статуси Дубідок".

### 8.2 Розділ "Платежі" (`/payments`)

**Список** з фільтрами:

- Статус (multi-select): `received`, `classified`, `awaiting_review`, `in_queue`, `skipped`
- Період (date range; default — поточний місяць)
- Клієнт (autocomplete по `Client.name` / `legal_id`)
- Тип послуги (`access` / `sms` / `other` / без типу)
- Сума (min/max)
- Текстовий пошук по `purpose`, `payer_name`

**Колонки списку:** дата, платник (name + legal_id), сума, призначення (truncated), статус (бейдж), reason (якщо in_queue/awaiting_review), клієнт (link на картку), акт (link якщо є).

**Картка платежу (`/payments/[id]`):**

- Загальна інформація: дата, сума, платник, призначення (повністю), `raw_data` (collapsible JSON).
- Зматчений клієнт (link) і `service_type`, `unit_price`, `quantity`, `parsed_contract_number`.
- Статус + reason (якщо є).
- Згенерований акт (link) — якщо `act_id IS NOT NULL`.
- **Дії**:
  - Якщо `status ∈ {awaiting_review, in_queue}`: "Класифікувати вручну" (форма з полями для корекції client / service_type / quantity → реруниться класифікація → генерується акт), "Пропустити" (status → `skipped`).
  - Якщо `status = received` (рідкісно — означає, що класифікатор не встиг): "Класифікувати зараз".
  - Якщо `status = classified`: read-only режим перегляду.
  - "Створити нового клієнта" — відкриває форму створення Client (8.4) з предзаповненими полями з `payer_name`, `payer_legal_id`, `payer_bank_account`.

### 8.3 Розділ "Черга розбору" (`/queue`)

**Чим відрізняється від `/payments?status=in_queue`:** це фокусована робоча область з groupings, швидкими діями і peer-set-фільтрами, орієнтована на task completion ("розібрати все за сесію"), а не на огляд.

**Дві вкладки:**

**Tab 1: "На апрув"** (`Payment.status = awaiting_review`):

- Reasons (D-014, D-036): `auto_act_disabled` (клієнт у списку виключень), `external_edo` (клієнт оформлюється через Вчасно). Картки візуально розрізняються: `external_edo` має бейдж "Вчасно" і підказку "Після генерації PDF — завантажте у Вчасно і поверніться позначити підписаним".
- Group by `classification_reason` (підсекції в межах вкладки).
- Список з картками: дата, клієнт, сума, призначення; кнопки "Згенерувати акт" (з можливістю редагування параметрів перед генерацією), "Пропустити".
- Для `external_edo` кнопка "Згенерувати акт" створює акт з `edo_provider = vchasno_external` і НЕ викликає Дубідок API (D-036, 7.4).

**Tab 2: "Проблеми класифікації"** (`Payment.status = in_queue`):

- Group by `classification_reason`. Кожна група згортається; у заголовку — кількість.
- Reasons (за частотою інцидентів):
  - `client_incomplete` (картка показує missing-список з посиланнями на поля клієнта для швидкого заповнення)
  - `no_match` (швидкі дії: "Створити нового клієнта", "Прив'язати до існуючого")
  - `multiple_contracts` (D-027 — список знайдених номерів з radio-select → перекласифікація)
  - `ambiguous_client` (показує client.legal_id і payer.legal_id; "Підтвердити клієнта", "Створити іншого")
  - `amount_mismatch` (показує amount, unit_price, очікувані quantity варіанти)
  - `sms_quantity_mismatch` (показує parsed_quantity і amount/unit_price; форма корекції)
  - `unknown_service_type` (резерв — поки не виникає)

**Швидкі дії в межах queue** (без переходу на картку платежу):

- Inline-форма корекції найчастіших полів (клієнт, quantity).
- "Пропустити" — терминальний skipped.
- "Відкрити картку" — повний вид.

- **Credentials в env**: `ADMIN_EMAIL`, `ADMIN_PASSWORD_HASH` (argon2id). Окремої таблиці User немає — single admin (D-001).

- **TLS обов'язковий** у production (10.3).
- **Logout** видаляє session-row і clears cookie.
- **Rate limiting** на login — простий counter у тій же session-таблиці (наприклад, max 10 спроб/година з одного IP). Деталь 10.3.

## 9. Edge cases і обробка помилок

Цей розділ перераховує конкретні ситуації, які виходять за рамки "happy path" класифікації і потребують або захисту в коді, або явного UX-флоу. Більшість покрита прийнятими рішеннями (D-001…D-033) — нижче зведення з посиланнями.

### 9.1 Дубль номера договору

**Кейс:** платіж має `purpose` з номером договору, який є в нашій БД, але `payer_legal_id` не збігається з `Client.legal_id` цього договору. Виявлено 8 таких випадків у виписці (`research/payments-analysis.md`).

**Поведінка:** `Payment.status = in_queue`, `classification_reason = ambiguous_client` (D-009).

**UX:** у черзі 8.3 (Tab "Проблеми класифікації", group `ambiguous_client`) платіж показує знайденого клієнта і ЄДРПОУ платника поруч; адмін підтверджує одного з двох (або створює нового клієнта).

### 9.2 Платіж від невідомого контрагента

**Кейс:** `parsed_contract_numbers` порожній (або номер не зматчено) І `payer_legal_id` не знайдено в `Client.legal_id`.

**Поведінка:** `Payment.status = in_queue`, `classification_reason = no_match`.

**UX:** у 8.3 з group `no_match` адмін бачить призначення і реквізити платника; дві швидкі дії: "Створити нового клієнта" (форма передзаповнена з `payer_*`) або "Прив'язати до існуючого" (autocomplete пошук). Після прив'язки/створення — реруниться класифікація.

### 9.3 Сума не сходиться з ціною

**Кейс access:** `payment.amount % unit_price != 0`. Наприклад, ціна 200, оплата 350 — не ділиться націло.

**Кейс sms:** `parsed_quantity × unit_price != payment.amount`. Наприклад, `"у кількості 100"` і ціна 1.40 → очікувано 140 грн, а прийшло 150.

**Поведінка:** `in_queue` з reason `amount_mismatch` (access) або `sms_quantity_mismatch` (sms).

**UX (8.3):**

- Для `amount_mismatch`: показуємо amount, unit_price і список можливих варіантів: "350 / 200 = 1.75 (не валідно)", "1 міс × 200 = 200 (надлишок 150)", "2 міс × 200 = 400 (недоплата 50)". Адмін: коректує `unit_price` через override клієнта, виставляє service_type=other з ручним unit_price, або skipped.
- Для `sms_quantity_mismatch`: показуємо `parsed_quantity`, `amount/unit_price` (calculated). Адмін править quantity або позначає other.

### 9.4 Транзитний рахунок без номера договору

**Кейс:** платіж приходить через транзитний рахунок ПриватБанку (`payer_legal_id ∈ Settings.transit_edrpou_list`), і в призначенні немає номера договору. Реальний платник — фізособа з ПІБ після коми, але fuzzy-матч по ПІБ не реалізовано (D-008).

**Поведінка:** `in_queue` з reason `no_match` (для транзиту lookup тільки по contract_number; якщо немає — fallback не спрацьовує).

**UX:** як у 9.2. Можна додати в адмінку підказку "Платіж з транзитного рахунку — реальний платник у тексті після коми". Fuzzy-матч по ПІБ як підказка — Phase 2.

### 9.5 Зміна реквізитів клієнта після генерації акту

**Кейс:** після того, як на клієнта згенеровано один або кілька актів, він змінив `address`, `bank_account`, `email` або іншу інфу. Sync з "Моє ОСББ" (або ручне редагування) оновлює `Client.*` поля.

**Поведінка:** `Act.client_snapshot` залишається з даними на момент генерації (D-004) — історичні акти **юридично незмінні**.

**UX:**

- Адмін бачить актуальні реквізити в картці клієнта (8.4) і snapshot-реквізити в картці акту (8.6).
- Якщо потрібно перевипустити старий акт із новими реквізитами (наприклад, помилка реквізитів критична для бухгалтерії клієнта): окремий ручний flow в 8.6 — "Перевипустити акт" → старий акт у Дубідок архівують → новий генерується з поточних `Client.*`.

**Що НЕ робимо:** не пропонуємо bulk-перевипуск всіх історичних актів. Це окрема ручна дія, бо кожен старий акт уже підписаний і анулювати/перевипускати кожен — юридична операція.

### 9.6 Збій ПриватБанк API

**Кейси:**

- `401 Unauthorized` (токен невалідний/протух): polling зупиняється.
- `5xx` / network: 3 retries з backoff (1с / 5с / 30с). Якщо не виходить — пропускаємо цикл, наступний polling зчитає overlapping період.
- `429 Too Many Requests`: respect Retry-After.
- 4+ послідовних провали — банер у адмінці (D-030) з типом помилки і часом останньої успішної взаємодії.

**Втрата даних не загрозлива:** polling overlapping (~2 години при інтервалі 1 година — 7.1) + UNIQUE по `bank_transaction_id` — будь-який пропуск компенсується наступним успішним polling-ом без дублікатів (4.7).

**Адмін-дія:** перевірити токен (env), перевипустити при потребі (зміна env + рестарт процесу — D-020), потім кнопка "Синхронізувати зараз" з дашборда (8.1).

### 9.7 Збій Дубідок API при відправці акту

**Стосується тільки актів з `edo_provider = dubidoc`** (D-036). Для `vchasno_external` API-виклику немає — цей edge case не виникає.

**Кейси:**

- `POST /api/v1/documents` повертає 5xx або timeout: акт лишається у статусі `draft`, у моделі є запис, але в Дубідок його ще немає.
- `422 Validation Error` (наприклад, ЄДРПОУ некоректного формату): лог помилки в системі, банер.
- `401 Unauthorized`: токен невалідний, зупинка відправок.

**Поведінка:**

- 3 retries з backoff при 5xx/timeout.
- Якщо не виходить — `Act.status` лишається `draft`. У UI 8.6 акт з індикатором "Не відправлено" з кнопкою "Спробувати ще раз".
- Адмін може повторити вручну з картки акту або з дашборда (групова дія "Відправити всі draft").

**Idempotency при retry:** при повторній відправці робимо новий `POST /documents` — Дубідок не має ідемпотентного ключа в API. Захист: перевіряємо, чи `Act.edo_doc_id IS NULL` перед відправкою. Якщо є — це означає, що відправка вже відбулась успішно, retry не потрібен.

### 9.8 Повторна обробка одного й того ж платежу

**Кейс:** polling-цикли overlapping, або процес перезапустився під час класифікації.

**Поведінка:** покрито idempotency на двох рівнях (4.7):

- `INSERT ... ON CONFLICT (bank_transaction_id) DO NOTHING` на рівні Payment — дублікат на запис.
- Класифікація працює тільки на `status=received`; обгорнута в транзакцію з `SELECT ... FOR UPDATE` — паралельні запуски однієї транзакції чекають.

**Якщо акт уже згенеровано (`Payment.act_id IS NOT NULL`):** повторна класифікація для цього платежу не запускається; ручний reclassify заблокований на рівні UI (форма недоступна для `status=classified`).

### 9.9 Видалення акту в Дубідок (зворотний sync)

**Стосується тільки актів з `edo_provider = dubidoc`** (D-036). Для `vchasno_external` ні запису в Дубідок немає, ні polling-а — еквівалентний кейс (видалення документа у Вчасно) система не виявляє і не обробляє автоматично; адмін за потреби вручну змінює `status` через картку акту (8.6, "Скасувати позначку підпису").

**Кейс:** адмін у Дубідок UI вручну видалив/архівував документ (наприклад, помилково створений). API Дубідок не дає webhook на цю подію (D-029 — webhook'и взагалі не використовуємо; polling це покриває).

**Поведінка:** наступний polling-цикл (раз в `dubidoc_poll_interval_hours`, default 6) `GET /api/v1/documents/{id}` поверне `archived = true` для цього `Act`.

**Дії системи:**

- `Act.status = deleted` (D-029).
- `Payment.act_id = NULL` (через `ON DELETE SET NULL` — D-025; але тут ми не видаляємо Act, а змінюємо status — fallback `Act.status` модельний; explicit `UPDATE payments SET act_id = NULL WHERE act_id = $deleted_act.id` у polling-handler).
- Платіж стає доступним для повторної класифікації. У UI 8.2 платіж знову з'являється з підказкою "Попередній акт видалений у Дубідок, можна перекласифікувати".

**Адмін-flow:** заходить у UI, бачить платіж знову у списку, або реруниться класифікація (якщо дані не змінились — створиться новий акт з наступним номером в серії клієнта).

### 9.10 Зміна ціни в період відкладеного платежу

**Кейс:** платіж отримано 1 квітня, але класифікація відкладена (наприклад, був `in_queue(client_incomplete)`). Адмін розбирався тиждень. За цей час адмін підняв `unit_price` для access або `SmsPrice` для sms — змінилась тарифна сітка.

**Поведінка:** `resolve_unit_price(client, service_type, payment_date)` (5.4) **бере ціну на дату ПЛАТЕЖУ**, не дату класифікації. Тариф і SmsPrice версіоновані через `effective_from` (5.1, 5.2, D-013, D-018, D-026), `tariff_lookup` / `resolve_sms_price` фільтрує по `effective_from <= payment_date`.

**Результат:** акт фіксує ціну на дату факту оплати — юридично коректно.

**Винятки:**

- Якщо `effective_from` нової ціни <= `payment_date` (тобто адмін заднім числом змінив ціну, що діє на момент платежу): нова ціна спрацює. Це навмисний дизайн — нова версія "переписує" історію резолву, але вже згенеровані акти захищені snapshot-ом (5.5).
- Override клієнта (`Client.access_price_override`) НЕ версіонується (5.3) — використовується поточне значення. Це навмисний компроміс: для override випадків (виняткові домовленості) історія змін малоймовірна; якщо потрібно — додаємо `OverrideHistory` окремим рішенням.

### 10.2 Performance

Орієнтири, що мають триматись у нормальному режимі (не SLA-контракт, а design target):

| Метрика                                                          | Target                         | Чому                                                     |
| ---------------------------------------------------------------- | ------------------------------ | -------------------------------------------------------- |
| Час від polling-події ПриватБанку до запису `Payment` у БД       | < 5 c                          | UNIQUE INSERT — швидко на Neon                           |
| Класифікація одного платежу (happy path → акт у БД)              | < 2 c                          | Включає resolve тарифу, генерацію номера, snapshot       |
| Рендер PDF одного акту (включно з cold start Chromium на Vercel) | < 8 c (cold), < 2 c (warm)     | Acceptable для async-у; UI не блокується                 |
| Відправка акту в Дубідок (`POST /documents`)                     | < 3 c (network round-trip)     | Залежить від API Дубідок                                 |
| End-to-end: polling → акт у Дубідок                              | < 60 c у happy path            | На практиці більшість часу — затримка між cron-тиками    |
| Polling статусу Дубідок (один запит на акт)                      | < 1 c                          | Batch не використовуємо — `GET /documents/{id}` поштучно |
| Адмін-сторінка (server-rendered)                                 | < 1.5 c first contentful paint | Next.js App Router з Server Components                   |

**SLA polling-у:** не строгий — overlapping period (~2× інтервал, D-029, 7.1) забезпечує eventual consistency. Допускається пропуск окремих циклів через 5xx-збої зовнішніх API без втрати даних.

**Масштаб (для контексту, не для оптимізації):**

- ~300 клієнтів, ~500 платежів/місяць, ~500-600 актів/місяць.
- Sync "Моє ОСББ" — раз на місяць (D-023), ~300 рядків.
- Жодних оптимізацій під більший масштаб у MVP не робимо. Якщо колись стане 3000 клієнтів — повертаємось до оптимізацій.

### 10.3 Безпека

**Secrets (D-020, D-034):**

- Усі credentials і ключі — **тільки в env**, ніколи в БД, ніколи в коді.
- Перелік обов'язкових env-змінних:
  - `PRIVATBANK_TOKEN` — токен ПриватБанк Автоклієнт API
  - `DUBIDOC_TOKEN` — токен Дубідок API
  - `MOEOSBB_DB_URL` — connection string MySQL "Моє ОСББ" (формат `mysql://user:pass@host:port/db`)
  - `ADMIN_EMAIL` — email адміна для login
  - `ADMIN_PASSWORD_HASH` — argon2id-хеш паролю (D-032)
  - `SESSION_SECRET` — HMAC секрет для підпису session-токенів
  - `POSTGRES_URL` / `BLOB_READ_WRITE_TOKEN` — auto-provisioned Vercel Marketplace (Neon / Vercel Blob)
- Управління через `vercel env` (preview / development / production окремо).
- **Не логуємо значення env** (`PRIVATBANK_TOKEN=***` у логах). У structured-логах будь-яке поле з ключем "_token_", "_password_", "_secret_", "_url_" повинно бути redacted.
- **Не показуємо токени в UI 8.7**; тільки read-only статус підключення (D-020).

**Доступ до БД:**

- Postgres: `POSTGRES_URL` тільки у Vercel env; локально — `.env.local` через `vercel env pull` (не комітиться).
- MySQL "Моє ОСББ": окремий **read-only MySQL-користувач** (GRANT SELECT only), connection string в env (7.3, D-034). Деталі мережевого доступу з Vercel — вирішуються при імплементації (D-034 — TBD).

**TLS:**

- Production: обов'язково (D-032). Vercel забезпечує auto TLS (D-034).
- Dev (localhost): HTTP допускається.

**Auth і session (D-032):**

- Пароль зберігається тільки як argon2id хеш в env.
- Session-токен: HMAC-підписаний випадковий 256-bit secret, expiration 30 днів, stored у `session` таблиці (token_hash, expires_at, created_at).
- Cookie: HTTP-only, Secure, SameSite=Lax.
- Rate-limit на login: max 10 спроб/година з одного IP (counter у тій же `session`-таблиці або окремій). Захист від brute-force.
- Logout видаляє session-row і clears cookie.

**Резервне копіювання:**

- Neon Postgres має **автоматичні point-in-time бекапи** (PITR) на безкоштовному tier-і — 7 днів історії. Pro-tier дає більше. У продукції — увімкнути.
- Додатково: щотижневий `pg_dump` через Vercel Cron у Vercel Blob (зашифрований password-based, ключ в env). Деталь Phase 1.
- Vercel Blob (PDF файли): нативного backup немає, але PDF-и регенерувальні з БД через шаблони (snapshot-полів `Act.*` достатньо для повторного рендеру).

**Інше:**

- Жодних write-операцій у БД "Моє ОСББ" (D-001).
- ON DELETE RESTRICT на історичних сутностях (D-025) — БД-рівнева гарантія від втрати юридичних документів.

### 10.4 Логування і моніторинг

**Формат логів:**

- **Structured JSON у stdout** (pino або еквівалент). Vercel автоматично збирає stdout в Vercel Logs (queryable з UI/CLI).
- Поля log-event-а: `timestamp`, `level`, `event` (короткий машинно-читабельний ключ типу `payment.classified`, `dubidoc.post.failed`), `payment_id`/`client_id`/`act_id` (де релевантно), `details` (object).
- Sensitive поля (token, password, url з credentials) — redact на рівні логера.

**Що логуємо обов'язково:**

- Усі state-transitions: `Payment.status` зміни, `Act.status` зміни.
- Усі виклики зовнішніх API: timestamp, endpoint, status code, latency. Тіло запиту/відповіді — тільки на `level=debug` (за прапором `DEBUG_API=true`), бо містить PII.
- Помилки класифікації з причиною (reason + missing-список для `client_incomplete`).
- Cron-задачі: початок/кінець, кількість оброблених елементів, кількість помилок.
- Login-події: success/failure, IP (для rate-limit), email (без password).

**Алерти (D-030):**

- Тільки **банер у адмінці** на дашборді 8.1. Без push-каналів у MVP.
- Дані банера — з runtime-таблиці `integration_health` (не моделюється в `domain-model.md`, ad-hoc): last_success_at, last_error_at, last_error_code, last_error_message для кожного зовнішнього сервісу (PrivatBank, Dubidoc, Moeosbb).
- Phase 2: можна додати Telegram-bot або Sentry (D-030 alternatives).

**Що НЕ логуємо:**

- Sensitive values (токени, паролі, повні connection strings з credentials).
- Тіло webhook-ів (D-029: webhook не використовуємо).
- Health-check pings (без винятку — заглушать сигнал).

## 11. Етапи реалізації (roadmap)

Roadmap побудована за принципом "запустити end-to-end якомога швидше, потім ітеративно покращувати UX і edge cases". Кожен етап завершується станом, у якому система **жива** (генерує реальні акти на реальних платежах), не пів-готова.

### Phase 0: MVP — end-to-end happy path + черга розбору

**Мета:** автор перестає виставляти більшість актів вручну. Система покриває happy path і всі основні non-happy шляхи через чергу розбору.

**Backend pipeline:**

- Polling ПриватБанк Автоклієнт API раз на годину (D-024 PostgreSQL, реюз коду з privatbank-telegram-bot — 7.1).

- Резолв ціни через Tariff (D-013, D-018) і SmsPrice (D-026) з override клієнта (D-017).
- Snapshot всіх immutable полів акту (D-010, D-019).
- Генерація PDF локально через HTML+React+Tailwind→headless browser (D-028); зразки шаблонів для access і sms.
- Відправка акту в Дубідок (`POST /api/v1/documents`) з inline participants (D-029) — для клієнтів з `edo_provider = dubidoc`.

**Адмін-інтерфейс (Next.js + Tailwind + shadcn — D-031):**

- Дашборд (8.1): банери стану зовнішніх API, лічильники, останні 10 платежів, кнопки sync.

- Перевипуск акту як explicit flow в 8.6 (9.5).

### Phase 2: push-канали, оптимізації

**Тригер переходу:** Phase 1 покрила найбільш частий feedback; нагромадились бажалки, які не блокують роботу але покращують зручність.

**Можливі напрямки** (обирається з фактичних потреб):

- Push-алерти (Telegram bot / email) для критичних інцидентів (D-030 alternatives).
- Webhook від Дубідок для real-time статусів (заміна / додаток до polling).
- Fuzzy-матч по ПІБ для транзитних рахунків як підказка адміну (D-008).
- 2FA на login (D-032 alternatives).
- Розширена історія — `OverrideHistory` для версіонування `Client.access_price_override` (9.10).
- Аудит-лог дій адміна (хто коли видалив договір тощо).

### Phase 3+ (за потреби, не плануємо явно зараз)

Все, що з'явиться як реальна потреба, але не вписується в чітку часову рамку Phase 1/2:

- **Повноцінна API-інтеграція з Вчасно** (зараз `vchasno_external` — ручний канал без API, D-036). Тригер: помітна частка клієнтів через Вчасно і ручний процес стає тягарем.
- Інтеграція з третім EDO-провайдером (D-015 закладає бренд-нейтральні поля `edo_*`; enum `edo_provider` розширюється новим значенням).
- Multi-admin (D-001 — single admin), якщо колись стане потрібно делегувати частину роботи.
- Розширена аналітика для податкових цілей (D-001 — звітність явно out of scope).

**Приклад payload із ПриватБанку:**
Точний формат залежить від використовуваного endpoint Автоклієнт API; повний референс — `api-docs/Privatbank_API.pdf`. Ключові поля, які мапимо на `Payment` (7.1):

```json
{
  "id": "PB12345abcde", // → bank_transaction_id (UNIQUE)
  "date": "2026-04-05", // → payment_date
  "amount": "200.00", // → amount
  "purpose": "За надання доступу до сервісу Моє ОСББ, договір №556770. Без ПДВ",
  "payer": {
    "name": "ОСББ «Приклад»", // → payer_name
    "legal_id": "12345678", // → payer_legal_id
    "iban": "UA12..." // → payer_bank_account
  }
}
```

Повний payload зберігається в `Payment.raw_data` (jsonb) для аудиту.

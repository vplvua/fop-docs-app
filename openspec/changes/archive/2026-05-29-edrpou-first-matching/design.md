## Context

Поточний матчинг (`lib/classification/match-client.ts`, спек `classification` → «Client matching — two-factor verification», ADR D-009) пріоритезує номер договору з призначення, перевіряючи ЄДРПОУ платника лише як другий фактор. Це створює клас «тихих» неправильних класифікацій: платник вписує хибний номер договору, цей номер унікально вказує на якийсь запис клієнта, ЄДРПОУ збігається (бо це дублікат тієї ж компанії) — і платіж тихо привʼязується до неправильного запису.

Незмінні обмеження домену:

- **Дублікати клієнтів невидаляльні.** Синк Моє ОСББ матчить записи по `moeosbb_user_id` й відновлює видалене; очищення ЄДРПОУ перезаписується при синку; `contract.number` — обовʼязкове поле. Тому система мусить толерувати кілька постійних записів з одним ЄДРПОУ.
- **`auto_act_disabled` = «актів не роблю».** Оператор позначає так і сміттєві дублікати, і exclusion-list клієнтів (D-003, Урбан/Авалон). Прапорець переживає синк (manual-only поле). Вкладка «Архів» у списку клієнтів = `auto_act_disabled = true`.
- **ЄДРПОУ платника надійний**, номер договору — ні (вводить платник).
- **БД на Neon, Drizzle.** Enum `classification_reason` — Postgres ENUM; нове значення додається через `ALTER TYPE ... ADD VALUE`.

Зачеплені ADR: **D-009** (переглядається), **D-008** (транзит — зберігається), **D-014** (awaiting_review/auto_act_disabled — зберігається), **D-027** (multiple_contracts — уточнюється), **D-003** (exclusion-list — зберігається).

## Goals / Non-Goals

**Goals:**

- Зробити ЄДРПОУ платника первинним фактором ідентифікації клієнта; договір — вторинним розрізнювачем у межах ЄДРПОУ.
- Прибрати тихі неправильні класифікації: коли ЄДРПОУ має одного активного клієнта — платіж іде до нього незалежно від (хибного) номера договору.
- Зробити неоднозначність видимою: кілька активних клієнтів з одним ЄДРПОУ → явне `awaiting_review` з селектором лише активних.
- Зберегти exclusion-list (D-003/D-014) і транзит (D-008).

**Non-Goals:**

- Дедуплікація/мердж клієнтів — поза скоупом (дублікати лишаються, ми лише не даємо їм перемагати матчинг).
- Розщеплення одного платежу на кілька актів (лишається D-007: 1 платіж = 1 акт; `multiple_contracts` як був).
- Зміна семантики архівування/розархівування клієнта — використовуємо наявний `auto_act_disabled`.
- Зміна сценарію транзитних платежів по суті.

## Decisions

### D1. Порядок матчингу: ЄДРПОУ-first (переглядає D-009)

Для **не-транзитних** платежів `matchClient` визначає клієнта так:

1. `candidates` = клієнти з `legal_id == payer_legal_id` (активні + архівовані).
2. `activeCandidates` = `candidates` з `auto_act_disabled == false`.
3. Резолв:
   - `candidates.length == 0` → `in_queue(no_match)`.
   - `activeCandidates.length == 1` → **matched** (далі звичайний pipeline → акт).
   - `activeCandidates.length > 1` → розрізнення по договору серед `activeCandidates`:
     - рівно 1 активний кандидат має `contract.number ∈ parsed_contract_numbers` → **matched**;
     - інакше → `awaiting_review(multiple_clients_same_edrpou)`, у результат кладемо `candidateClientIds` = id активних кандидатів (для селектора).
   - `activeCandidates.length == 0` (є лише архівовані з цим ЄДРПОУ) → `awaiting_review(auto_act_disabled)`, `clientId` = архівований кандидат, якщо він один; якщо архівованих кілька — `clientId = null` (потрібне розархівування потрібного). Зберігає D-014/D-003 для Урбан/Авалон.

**Чому не «контракт-first як було»:** номер договору пише платник і помиляється; ЄДРПОУ — його власний надійний реквізит. ЄДРПОУ ніколи не вкаже на чужу компанію.

**Чому архівовані не у `activeCandidates`:** архів = «актів не роблю». Архівований запис не повинен ні авто-класифікуватись в акт, ні зʼявлятись у селекторі. Але він лишається у `candidates`, щоб коректно обробити випадок «лише архівовані» (Урбан/Авалон) і не зламати D-014.

**Альтернатива (відкинуто):** повністю виключати архівованих із матчингу. Зламало б exclusion-list (Урбан/Авалон → `no_match` замість `awaiting_review`).

**Альтернатива (відкинуто):** «завжди селектор при >1 кандидаті». Призвело б до ручного кліку на кожен платіж РЕСПЕКТ назавжди (5 дублікатів незнищенні). «Перевага єдиному активному» авто-вирішує цей кейс.

### D2. Транзит зберігається (D-008)

Якщо `payer_legal_id ∈ transit_edrpou_list` — матч по договору серед `activeCandidates` (без перевірки ЄДРПОУ, бо це ЄДРПОУ банку). Якщо договір вказує рівно одного активного → matched; кілька активних по договору → `awaiting_review(multiple_clients_same_edrpou)` з селектором; жодного активного → `in_queue(no_match)` (або `auto_act_disabled`, якщо знайдено лише архівованого по договору).

### D3. Новий reason `multiple_clients_same_edrpou` → `awaiting_review`

Семантично «клієнта-компанію впізнано, але є кілька активних записів — оберіть вручну». Лягає в `awaiting_review` (за D-014: впізнано, але авто-акт заблоковано до ручного кроку), а не в `in_queue` (то «не зрозуміли, хто платить»). Реалізація: `ALTER TYPE classification_reason ADD VALUE 'multiple_clients_same_edrpou'`.

### D4. Взаємодія з `multiple_contracts` (уточнює D-027)

`multiple_contracts` блокує лише коли договір реально є розрізнювачем. Порядок:

- Якщо ЄДРПОУ-резолв дає рівно 1 активного клієнта → parsed-номери інформативні, **не блокуємо** (навіть якщо їх кілька).
- Якщо потрібне розрізнення по договору (транзит або >1 активний з ЄДРПОУ) і в призначенні >1 різних номерів → лишається `in_queue(multiple_contracts)` як у D-027.

### D5. Інваріант ручної привʼязки (нова server action)

Ручний вибір/привʼязка платежу дозволена лише до клієнта з `legal_id == payment.payer_legal_id` (виняток — транзит: дозволені клієнти, на яких вказує договір). Server action валідує це й відхиляє вибір клієнта з іншим ЄДРПОУ. Після привʼязки запускається решта pipeline для обраного клієнта (service_type → completeness → price → act), з повагою до `auto_act_disabled`/`edo_provider`.

### D6. Форма результату матчингу

`MatchResult` розширюється: до `matched` / `in_queue` додається варіант, що несе `candidateClientIds: string[]` для `multiple_clients_same_edrpou`, щоб UI міг показати селектор без повторного запиту. `clientId` зберігається в платежі і для `awaiting_review`-станів (як уже робиться для `auto_act_disabled`).

## Risks / Trade-offs

- **[Зміна класифікації наявних платежів]** Платежі, де договір і ЄДРПОУ розходяться, тепер класифікуються інакше (раніше `ambiguous_client`/тихий вибір). → Mitigation: після деплою прогнати повторну класифікацію `in_queue`/`awaiting_review`; зміна не чіпає вже `classified` (їх не рекласифікують, FR-CLASS-17).
- **[Регрес exclusion-list]** Можна випадково зламати Урбан/Авалон (`auto_act_disabled` одиночний). → Mitigation: явний сценарій у спеці + unit-тест «єдиний архівований клієнт на ЄДРПОУ → awaiting_review(auto_act_disabled)».
- **[Кілька архівованих без активного]** `clientId = null`, оператор має розархівувати потрібного. → Mitigation: UI підказка в `auto_act_disabled`-стані, коли активного кандидата немає; покривається сценарієм.
- **[Транзит + дублікати]** Рідкісний кут; договір серед активних може дати кілька. → Mitigation: той самий шлях `multiple_clients_same_edrpou` + селектор.
- **[Postgres ENUM ADD VALUE]** `ALTER TYPE ADD VALUE` не відкочується в одній транзакції. → Mitigation: окрема міграція; dev/prod гілки Neon мігруються окремо (див. `docs/operations.md`).

## Migration Plan

1. Drizzle-міграція: `ALTER TYPE classification_reason ADD VALUE 'multiple_clients_same_edrpou'`. Застосувати на dev, потім окремо на prod (runbook).
2. Деплой коду (matcher + classify + UI).
3. Прогнати повторну класифікацію наявних `in_queue`/`awaiting_review` платежів (manual reclassify або одноразовий скрипт) — особливо РЕСПЕКТ.
4. Rollback: код відкочується звичайно; новий enum value лишається в типі (безпечно, не використовується старим кодом).

- Чи показувати в `auto_act_disabled`-стані список архівованих кандидатів з кнопкою «Розархівувати і класифікувати», коли активного немає? (зручно, але можна винести в окрему ітерацію).

## Why

Платник є єдиним надійним ідентифікатором компанії в платежі — його ЄДРПОУ. Номер договору в призначенні пише платник вручну і регулярно помиляється. Поточний матчинг (D-009) пріоритезує саме номер договору, тому хибний номер тихо привʼязує платіж до неправильного запису клієнта.

Реальний кейс: компанія «РЕСПЕКТ» має один активний клієнт (договір 557352) і кілька дублікатів-сміття від синку Моє ОСББ (557355–557359), які неможливо видалити (синк відновлює, ЄДРПОУ/ID перезаписуються, номер договору — обовʼязкове поле). Платник вказав 557355 — і платіж тихо сів на архівований дублікат замість активного клієнта. Жодного сигналу про помилку не було.

## What Changes

- **BREAKING (логіка матчингу):** пріоритет визначення клієнта змінюється з «договір → ЄДРПОУ» (D-009) на **«ЄДРПОУ платника → договір як вторинний розрізнювач»**. Номер договору тепер лише розрізняє кандидатів у межах одного ЄДРПОУ, ніколи не перевизначає компанію.
- **Архівовані клієнти (`auto_act_disabled = true`) виключаються з активної вибірки кандидатів.** Архів у системі = «актів не роблю». Коли той самий ЄДРПОУ має активного клієнта, архівовані дублікати ігноруються і не можуть «вкрасти» платіж.
- **Транзитні платники (D-008) лишаються на матчингу по договору** — там ЄДРПОУ платника належить банку-посереднику і не може ідентифікувати клієнта. Поведінка зберігається.
- **Новий `classification_reason`: `multiple_clients_same_edrpou`** (статус `awaiting_review`). Виникає, коли кілька **активних** клієнтів мають той самий ЄДРПОУ і договір не розрізнює їх однозначно. UI показує попередження + **селектор лише з активних** клієнтів цього ЄДРПОУ для ручного вибору. Архівовані в селектор не потрапляють (за потреби — є розархівування).
- **Інваріант ручного вибору/привʼязки:** платіж можна привʼязати лише до клієнта, чий `legal_id` дорівнює `payer_legal_id` платежу (не транзит). Не можна виставити акт компанії, відмінній від платника.
- **Збереження D-014/D-003:** одиничний архівований клієнт на ЄДРПОУ (exclusion-list Урбан/Авалон) і далі дає `awaiting_review(auto_act_disabled)` — не ламаємо.
- **Уточнення взаємодії з D-027 (`multiple_contracts`):** коли ЄДРПОУ однозначно вказує одного активного клієнта, наявність кількох номерів договору в призначенні більше не блокує класифікацію — номери стають інформативними. `multiple_contracts` лишається релевантним лише там, де договір реально використовується як розрізнювач (транзит або кілька активних з тим самим ЄДРПОУ).

### Modified Capabilities

- `classification`: змінюється «Client matching — two-factor verification» (пріоритет ЄДРПОУ-first, виключення архівованих з вибірки, розрізнення кількох активних по договору); додається новий reason `multiple_clients_same_edrpou` зі станом `awaiting_review`; уточнюється взаємодія з `multiple_contracts` та `auto_act_disabled`; додається інваріант «привʼязка лише в межах ЄДРПОУ».
- `payments-ingest`: на сторінці/в UI платежу — попередження про кілька активних клієнтів з одним ЄДРПОУ та селектор вибору активного клієнта (ручна привʼязка в межах ЄДРПОУ).

## Impact

- **Код:** `lib/classification/match-client.ts` (переписується порядок матчингу), `lib/classification/classify.ts` (крок матчингу + взаємодія з multiple_contracts/auto_act_disabled), `lib/classification/types.ts` (новий reason + результат із кандидатами), `lib/classification/run-classification.ts` (можливо ручна привʼязка/повторна класифікація з обраним клієнтом), `app/(dashboard)/payments/[id]/` (панель класифікації: попередження + селектор, нова server action для привʼязки в межах ЄДРПОУ).
- **БД:** новий enum value `classification_reason = 'multiple_clients_same_edrpou'` (`ALTER TYPE ... ADD VALUE`), міграція через drizzle-kit. Окремих нових таблиць/колонок не потрібно.
- **ADR:** новий ADR (напр. D-041) фіксує EDRPOU-first і **переглядає D-009**; уточнює межі D-008, D-014, D-027, D-003.
- **Тести:** unit на `match-client` (нові гілки), integration smoke на класифікацію кількох активних/архівованих з одним ЄДРПОУ.
- **Сумісність:** змінюється результат класифікації для платежів, де номер договору і ЄДРПОУ розходяться — раніше `ambiguous_client`/тихий вибір, тепер ЄДРПОУ-first. Потрібен прохід по наявних `in_queue/awaiting_review` платежах (повторна класифікація).
